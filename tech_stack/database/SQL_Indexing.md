# Индексы в реляционных БД

**Индекс (`Index`)** — вспомогательная структура данных для ускорения поиска строк по значениям колонок.

### Зачем нужны индексы

- Ускорение операций чтения: фильтрация (`WHERE`), сортировка (`ORDER BY`), объединение таблиц (`JOIN`)
- Снижение нагрузки на ввод-вывод (`I/O`): СУБД читает меньше страниц с диска
- Обеспечение целостности: уникальные индексы (`Unique Index`) гарантируют отсутствие дубликатов

### Базовая механика работы

- Данные в индексе хранятся в отсортированном виде
- Поиск выполняется за логарифмическое время: O(log n) вместо O(n) при полном сканировании (Full Table Scan)
- Индекс хранит значения ключевых колонок (`key columns`) + ссылки на физические строки (`row pointers`) или на кластерный
  ключ (`clustered key`)

### Основные типы индексов

**B-Tree Index / Сбалансированный древовидный индекс**

- Стандартный тип по умолчанию в большинстве СУБД (PostgreSQL, MySQL, Oracle)
- Поддерживает: точное совпадение (=), диапазоны (>, <, BETWEEN), сортировку (ORDER BY), префиксный поиск (LIKE '
  prefix%')
- Механика: данные организованы в сбалансированное дерево, где каждый узел содержит отсортированные ключи и указатели на
  дочерние узлы

**Hash Index / Хеш-индекс**

- Поддерживает только точное совпадение (=)
- Механика: хеш-функция преобразует значение ключа в адрес ячейки; поиск за константное время O(1)
- Ограничения: не поддерживает диапазоны, сортировку, частичные совпадения

**Bitmap Index / Битовый индекс**

- Оптимален для колонок с низкой кардинальностью (`low cardinality`): пол, статус, флаг
- Механика: для каждого уникального значения хранится битовая маска, где бит указывает на наличие строки
- Часто используется в аналитических СУБД (Oracle, ClickHouse)

**GiST / GIN Indexes / Индексы для сложных данных**

- GiST (Generalized Search Tree): гибкая структура для гео-данных, полнотекстового поиска, диапазонов
- GIN (Generalized Inverted Index): эффективен для массивов, JSON, полнотекстового поиска
- Механика: позволяют определять пользовательские стратегии поиска и сравнения

### Классификация по структуре хранения

**Clustered Index / Кластерный индекс**

- Определяет физический порядок хранения данных в таблице
- На таблицу может быть только один кластерный индекс (часто это Primary Key)
- Данные на диске отсортированы согласно ключу индекса

**Non-Clustered Index / Некластерный индекс**

- Отдельная структура, хранящая отсортированные ключи + указатели на данные
- На таблицу можно создать множество некластерных индексов
- Указатель ведёт либо на физическую строку (RID), либо на ключ кластерного индекса

### Специальные виды индексов

**Unique Index / Уникальный индекс**

- Гарантирует, что все значения в индексируемой колонке (или комбинации колонок) уникальны
- Используется для реализации ограничений уникальности (UNIQUE constraint)

**Composite Index / Составной индекс**

- Индекс по нескольким колонкам: `CREATE INDEX idx_name ON table (col1, col2, col3)`
- Работает по правилу левого префикса (`Leftmost Prefix Rule`): индекс (`A`, `B`, `C`) эффективен для поиска по `A`, `A, B`, `A,
  B, C`, но не по `B` или `C` отдельно
- Порядок колонок в определении индекса критически важен для производительности

**Covering Index / Покрывающий индекс**

- Индекс, содержащий все колонки, требуемые запросом (в ключе или в INCLUDE-части)
- Позволяет выполнить запрос, обращаясь только к индексу, без перехода к таблице (`Index-Only Scan`)
- Пример: для запроса `SELECT id, name FROM users WHERE email = '...'` покрывающим будет индекс `(email) INCLUDE (id, name)`

### Индексы для PRIMARY KEY и UNIQUE ограничений

**Автоматическое создание индексов (Automatic Index Creation)**
- При определении ограничения `PRIMARY KEY` или `UNIQUE` СУБД автоматически создаёт соответствующий индекс для обеспечения целостности данных (data integrity)
- Не нужно вручную создавать индекс на колонку, которая уже является первичным ключом или имеет `UNIQUE constraint`
- Пример: `CREATE TABLE users (id SERIAL PRIMARY KEY, email VARCHAR(255) UNIQUE);` — индексы по id и email создадутся автоматически

**Как проверить существующие индексы**
- PostgreSQL: `\d table_name` или `SELECT indexname, indexdef FROM pg_indexes WHERE tablename = 'users';`
- MySQL: `SHOW INDEX FROM users;`
- MS SQL Server: `EXEC sp_helpindex 'users';`

**Важные нюансы**
- Автоматически созданный индекс получает системное имя (если не задано явно), что может усложнить поддержку и миграции
- Если нужен индекс с конкретными параметрами (`fillfactor`, `tablespace`, INCLUDE-колонки), лучше создать его явно: 
  `CREATE UNIQUE INDEX idx_users_email ON users (email) INCLUDE (id, created_at)`;
- В некоторых СУБД можно отложить создание индекса для ограничения (`DEFERRABLE`), но это редко используется в высоконагруженных системах
- Удаление ограничения `PRIMARY KEY` / `UNIQUE` автоматически удаляет связанный с ним индекс

> **Правило**: не дублируйте индексы. Перед CREATE INDEX проверьте, не покрывает ли уже существующий индекс (в том числе созданный автоматически для ограничения) ваш запрос.

### Ключевые метрики эффективности

**Cardinality / Кардинальность** — количество уникальных значений в колонке. Высокая кардинальность (`UUID`, `email`) делает
индекс эффективным; низкая (пол, флаг) — часто нецелесообразна для B-Tree.

**Selectivity / Селективность** — доля строк таблицы, возвращаемых по условию. Индекс выгоден, когда запрос возвращает
малую часть данных (< 10–20%).

### Цена индекса: о чём важно помнить

- Индексы замедляют операции записи (`INSERT`, `UPDATE`, `DELETE`), так как требуют обновления структуры индекса
- Каждый индекс занимает место на диске и в оперативной памяти (буферный пул / buffer pool)
- Избыточное индексирование усложняет работу оптимизатора запросов (`query optimizer`) и может привести к выбору
  неоптимального плана выполнения

> **Правило**: создавайте индекс под конкретный запрос или паттерн доступа (`access pattern`), а не «на всякий случай».

### Плюсы и минусы использования индексов

**Преимущества (Benefits)**
- Быстрый поиск по условию (`WHERE`): сокращение времени выполнения запросов на фильтрацию
- Ускорение сортировки (`ORDER BY`): данные в индексе уже отсортированы, не требуется дополнительная сортировка (`filesort`)
- Оптимизация объединения таблиц (`JOIN`): индекс по внешнему ключу (`Foreign Key`) ускоряет поиск связанных строк
- Поддержка уникальности (Uniqueness): `Unique Index` предотвращает дублирование данных на уровне БД
- Возможность Index-Only Scan: если индекс покрывающий (`Covering`), СУБД может выполнить запрос без обращения к таблице

**Недостатки (Costs)**
- Замедление записи (Write Overhead): при `INSERT`, `UPDATE`, `DELETE` необходимо обновлять все затронутые индексы
- Потребление места (Storage Overhead): каждый индекс занимает место на диске и в буферном пуле (buffer pool)
- Накладные расходы оптимизатора (Optimizer Overhead): большое число индексов усложняет выбор плана выполнения, может привести к субоптимальному решению
- Фрагментация (Fragmentation): при частых изменениях данных индекс требует периодической реорганизации (`REINDEX`, `OPTIMIZE`)

### Чек-лист: когда создавать индекс

✅ Создавайте индекс, если:
- Колонка активно используется в фильтрации (`WHERE column = value`)
- Колонка участвует в объединении таблиц (`JOIN ... ON table.fk = other.id`) — особенно внешние ключи (`Foreign Keys`)
- Запросы часто сортируют результаты по колонке (`ORDER BY column`)
- Используется диапазонный поиск (`BETWEEN`, `>`, `<`) или префиксный поиск (`LIKE 'prefix%'`)
- Кардинальность (Cardinality) колонки высокая: много уникальных значений (`email`, `UUID`, `timestamp`)
- Требуется гарантировать уникальность данных (добавьте `UNIQUE Index`)

❌ Избегайте создания индекса, если:
- Таблица очень маленькая (десятки/сотни строк): Full Scan часто быстрее поиска по индексу
- Колонка часто обновляется (`high write frequency`): накладные расходы на перестроение индекса перевешивают выгоду от чтения
- Низкая кардинальность (low cardinality): пол (М/Ж), статус с 2–3 значениями — B-Tree индекс будет неэффективен (исключение: Bitmap Index в аналитических СУБД)
- В запросе применяются функции к колонке без функционального индекса: `WHERE UPPER(name) = '...'` не использует обычный индекс по name
- Запрос использует шаблон с подстановочным знаком в начале: `LIKE '%suffix'` — B-Tree не может ускорить такой поиск

### Анализ и отладка: как понять, работает ли индекс

**Команда `EXPLAIN` / `EXPLAIN ANALYZE`**
- Показывает план выполнения запроса (execution plan) без/с фактическим выполнением
- Ключевые метрики:
  - Cost: оценочная стоимость операции (ниже — лучше)
  - Rows: ожидаемое/фактическое число возвращаемых строк
  - Width: средний размер строки в байтах
  - Actual Time: реальное время выполнения (только для `EXPLAIN ANALYZE`)

**Типы сканирования в плане выполнения**
- `Seq Scan` / `Full Table Scan`: последовательный обход всей таблицы — часто признак отсутствия индекса или его неэффективности
- `Index Scan`: поиск по индексу с переходом к строкам таблицы
- `Index Only Scan`: запрос выполнен полностью по индексу, без обращения к таблице (идеальный случай для покрывающих индексов)
- `Bitmap Scan`: сначала строится битовая карта подходящих строк, затем извлекаются данные (эффективно при комбинировании условий)

**Пример анализа**
`EXPLAIN ANALYZE SELECT * FROM users u WHERE email = 'alex@example.com';`

Что смотреть в выводе:
- Если видите `Seq Scan` на большой таблице — вероятно, нужен индекс по `email`
- Если `Index Scan`, но `Actual Time` высокий — проверьте селективность (Selectivity) и статистику таблицы (`ANALYZE`)

**Поиск неиспользуемых индексов (Unused Indexes)**
- PostgreSQL:
  `SELECT indexname, idx_scan, idx_tup_read FROM pg_stat_user_indexes WHERE idx_scan = 0 AND indexname NOT LIKE '%_pkey';`

- MySQL (Performance Schema):
  `SELECT OBJECT_NAME, INDEX_NAME, COUNT_READ FROM performance_schema.table_io_waits_summary_by_index_usage WHERE COUNT_READ = 0;`

- MS SQL Server:
  `SELECT object_name(i.object_id) AS table_name, i.name AS index_name, s.user_seeks, s.user_scans FROM sys.indexes i LEFT JOIN sys.dm_db_index_usage_stats s ON i.object_id = s.object_id AND i.index_id = s.index_id WHERE s.user_seeks IS NULL AND s.user_scans IS NULL AND i.is_primary_key = 0;`

> Регулярно анализируйте использование индексов в продакшене: «мёртвые» индексы только замедляют запись и тратят место.

### Частые ошибки (Anti-Patterns)

❌ «Индекс на всякий случай»
- Создание индекса на каждую колонку «чтобы точно работало» приводит к деградации производительности записи и усложнению плана выполнения
- Решение: создавайте индекс под конкретный медленный запрос или паттерн доступа (access pattern)

❌ Неправильный порядок колонок в составном индексе (Composite Index)
- Индекс `(last_name, first_name)` НЕ ускорит поиск только по first_name из-за правила левого префикса (Leftmost Prefix Rule)
- Решение: ставьте в начало индекса колонку с высокой селективностью и ту, которая чаще всего используется в фильтрации

❌ Индексирование длинных текстовых полей без необходимости
- Индекс по `VARCHAR(500)` занимает много места и может быть менее эффективным, чем индекс по префиксу
- Решение: используйте индекс по префиксу (prefix index): `CREATE INDEX idx_name ON users (name(100))` или выносите полнотекстовый поиск в специализированные индексы (`GIN`, `FULLTEXT`)

❌ Ожидание, что индекс ускорит любой LIKE
- Запрос `LIKE '%keyword%'` (подстановка в начале) не может использовать обычный B-Tree индекс
- Решение: для полнотекстового поиска используйте `FULLTEXT Index`, `GIN` + `tsvector` (PostgreSQL) или внешние движки (Elasticsearch)

❌ Игнорирование статистики и фрагментации
- Устаревшая статистика (statistics) приводит к неверному выбору плана оптимизатором
- Решение: регулярно выполняйте `ANALYZE` / `UPDATE STATISTICS`; мониторьте фрагментацию индексов и выполняйте `REINDEX` при необходимости

❌ Функции и выражения в условии без функционального индекса
- Запрос `WHERE YEAR(created_at) = 2024` не использует индекс по `created_at`
- Решение: переписывайте условие на диапазон: WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01' или создавайте функциональный индекс (functional index): `CREATE INDEX idx_year ON users (YEAR(created_at))`

> **Золотое правило**: измеряйте, а не предполагайте. Всегда проверяйте влияние индекса через EXPLAIN ANALYZE и мониторинг реальных метрик (query latency, I/O, write throughput).
