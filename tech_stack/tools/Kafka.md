# Apache Kafka

> `Apache Kafka` - распределённая платформа потоковой передачи событий (`event streaming platform`),
> предназначенная для построения надёжных, масштабируемых и высокопроизводительных систем обмена сообщениями в реальном
> времени.

Kafka изначально была разработана в LinkedIn как система логирования, но быстро эволюционировала в универсальный
backbone для асинхронной коммуникации между сервисами, аналитики данных в реальном времени, интеграции систем и
event-driven архитектур.

В отличие от традиционных брокеров сообщений (например, `RabbitMQ`, `ActiveMQ`), Kafka:

- хранит сообщения на диске с возможностью повторного чтения,
- поддерживает горизонтальное масштабирование через партиционирование,
- гарантирует упорядоченность внутри партиции,
- обеспечивает высокую пропускную способность (десятки–сотни тысяч сообщений в секунду),
- ориентирована на длительное хранение данных (дни, недели, месяцы).

---

### Основные цели и области применения

- **Event-driven микросервисы**: декуплинг компонентов через асинхронные события.
- **Логирование и агрегация**: централизованная сборка логов с множества источников.
- **Потоковая аналитика**: обработка данных в реальном времени (в связке с `Kafka Streams` или `Flink`).
- **Интеграция систем**: замена ETL-процессов и шин данных.
- **Репликация данных между БД и хранилищами** (через `Kafka Connect`).

---

### Краткая история и экосистема

- **2011**: Kafka создана в LinkedIn.
- **2012**: передана в Apache Software Foundation.
- **Confluent**: компания, основанная создателями Kafka, развивает коммерческую платформу поверх Apache Kafka (Confluent
  Platform), включая Schema Registry, ksqlDB, REST Proxy и др.
- **`KRaft` (Kafka Raft Metadata mode)**: начиная с версии 3.3+, Kafka может работать без `ZooKeeper`, используя
  собственный
  протокол консенсуса на основе `Raft`.

> Раньше `Kafka` использовала `ZooKeeper` для управления кластером и выборов лидера, но это добавляло сложность. Начиная
> с версии 3.x `Kafka` переходит на собственный механизм `KRaft`, убирая зависимость от `ZooKeeper`, что упрощает
> архитектуру.

---

### Почему Kafka, а не RabbitMQ/JMS/ActiveMQ?

| Характеристика         | Kafka                          | Традиционные брокеры (RabbitMQ и др.)        |
|------------------------|--------------------------------|----------------------------------------------|
| Модель                 | Журнал событий (log-based)     | Очередь сообщений (queue-based)              |
| Хранение               | Долгосрочное (на диске)        | Краткосрочное (в памяти или на диске до ACK) |
| Пропускная способность | Очень высокая                  | Умеренная                                    |
| Упорядоченность        | Гарантируется внутри партиции  | Гарантируется в рамках очереди               |
| Повторное чтение       | Поддерживается                 | Обычно нет (после ACK — удаление)            |
| Использование          | Потоковая обработка, аналитика | Таск-ориентированные системы                 |

Kafka не заменяет все брокеры — она решает другие задачи. Выбор зависит от требований:

- если нужна простая очередь задач — подойдёт RabbitMQ;
- если требуется обработка потоков событий с возможностью повторного воспроизведения — Kafka.

# Архитектурные основы Kafka

---

## Producer → Broker → Consumer модель

Kafka следует классической модели публикации-подписки (`publish-subscribe`), но с важными особенностями:

- **Producer** (производитель) отправляет сообщения в **топик** (`topic`).
- **Broker** (брокер) — это сервер Kafka, который хранит данные и обслуживает запросы.
- **Consumer** (потребитель) читает сообщения из топика.

В отличие от очередей «точка-точка», в Kafka:

- Одно сообщение может быть прочитано **многими потребителями** (если они в разных `consumer group`).
- Сообщения **не удаляются после чтения**, а хранятся в течение заданного времени (`retention period`).

---

## Топики (Topics) и партиции (Partitions)

- **Топик** — логический канал для передачи сообщений (например, `user-activity`, `payment-events`).
- Каждый топик делится на **партиции** — упорядоченные, неизменяемые последовательности сообщений.
- Партиции позволяют **масштабировать** обработку: разные потребители могут читать разные партиции параллельно.
- Внутри одной партиции порядок сообщений **гарантирован**.
- Количество партиций задаётся при создании топика и может быть увеличено (но не уменьшено).

Пример:

Топик `orders` с 3 партициями:

```text
- Партиция 0: [msg1, msg4, msg7]
- Партиция 1: [msg2, msg5]
- Партиция 2: [msg3, msg6]
```

Визуализация структуры топика:

```text
Topic: orders
│
├── Partition 0
│  ├── offset 0 → {orderId: "A1", status: "created"}
│  ├── offset 1 → {orderId: "A4", status: "shipped"}
│  └── offset 2 → {orderId: "A7", status: "delivered"}
│
├── Partition 1
│  ├── offset 0 → {orderId: "A2", status: "created"}
│  └── offset 1 → {orderId: "A5", status: "shipped"}
│
└── Partition 2
   ├── offset 0 → {orderId: "A3", status: "created"}
   └── offset 1 → {orderId: "A6", status: "cancelled"}
```

---

## Смещения (Offsets) и их роль

> `Offset` — это индекс в логе
>
> Можно представить партицию как массив (список). `Offset` — это индекс элемента в этом массиве. Первое сообщение имеет
> offset 0, второе — 1 и так далее. Это просто число, которое уникально идентифицирует сообщение внутри конкретной
> партиции.
>
> Когда Consumer читает сообщение, он не удаляет его. Вместо этого он коммитит offset в специальный внутренний топик
> Kafka (`__consumer_offsets`).
>
> Kafka запоминает:
>
> Для группы `order-processor-group`, топика `orders`, партиции `0`, последний прочитанный `offset` = `5`.
>
> Если `Consumer` упадет и перезапустится, он спросит у `Kafka`: "С какого места мне начать читать для моей группы?" и
> получит ответ: "С 6-го сообщения (`offset 5` уже прочитан)".

- Каждое сообщение в партиции имеет уникальный **offset** — целое число, возрастающее с каждым новым сообщением.
- Offset определяет позицию сообщения в логе партиции.
- Потребитель **отслеживает свой текущий offset**, чтобы знать, какие сообщения уже прочитаны.
- Offset сохраняется либо автоматически (`enable.auto.commit=true`), либо вручную (`consumer.commitSync()` /
  `commitAsync()`).

Разница между `auto commit` и `manual commit`

- Auto (`enable.auto.commit=true`):
    - `Consumer` раз в N секунд (по умолчанию 5) говорит: "Я прочитал всё, что получил до этого момента".
    - Если `Consumer` получил пачку сообщений, обработал 90%, но упал за секунду до авто-коммита, после перезапуска
      он начнет читать с последнего закоммиченного `offset`. Те 10% сообщений будут прочитаны повторно (`at-least-once`
      семантика).

- Manual (`commitSync`/ `Async`):
    - Ты сам контролируешь момент. Например, обработал сообщение из БД, сохранил результат, только потом закоммитил
      `offset`.
    - Если ты забыл вызвать коммит, `Consumer` никогда не сдвинет указатель.

> **Ключевое свойство**
>
> **`Offset` привязан к тройке: (`consumer-group` + `topic` + `partition`).**

- Группа A может читать партицию 0 с offset = 10 (обработала 10 сообщений).
- Группа B читает ту же партицию 0 с offset = 5 (она медленнее или позже подключилась).

Это позволяет разным сервисам (группам) читать один топик независимо, каждый со своей скоростью.

## Репликация и отказоустойчивость

> Зачем нужны реплики?
> В распределенной системе серверы (брокеры) падают. Если хранить данные только на одном брокере, при его потере
> данные исчезнут навсегда. Реплики — это копии партиции на разных физических машинах. Если один брокер умрет, данные
> останутся на другом.

- Каждая партиция может иметь несколько **реплик** (копий), распределённых по разным брокерам.
- Одна реплика является **leader**, остальные — **followers**.
- Все записи и чтения происходят через **leader**.
- Если брокер с leader выходит из строя, один из followers становится новым leader (автоматически).
- Это обеспечивает **высокую доступность** и защиту от потери данных.

---

## Leader и Follower реплики

> Если представить партицию как документ в Google Docs:
>
> `Leader` — это владелец документа, у которого открыт доступ на редактирование. Только он может менять содержимое.
>
> `Followers` — это читатели с доступом "только просмотр". Они постоянно видят обновления от лидера.

- **Leader**: принимает все записи от `producer`-ов и обслуживает запросы от `consumer`-ов.
- **Follower**: асинхронно копирует данные от `leader`-а.
- Только **`in-sync replicas` (`ISR`)** участвуют в выборе нового `leader`-а.

> Продюсер всегда пишет только в `Leader`. Если писать во все реплики сразу, нужно синхронизировать ответы — это
> медленно. Лидер принимает данные и рассылает их фолловерам.

---

## Кворум и подтверждения (`acks`)

Продюсер может настроить уровень надежности (`acks`):

- `acks=0`: Отправил и забыл. Быстро, но можно потерять данные.
- `acks=1`: Лидер записал данные себе и ответил "Ок". Если лидер упадет до того, как фолловеры скопировали данные, они
  потеряются.
- `acks=all` (`-1`): Лидер ждет подтверждения от всех in-sync реплик, что они получили данные. Медленнее, но надежнее.

> Кворум в действии:
>
> Когда кластер из 6 брокеров, а фактор репликации = 3, партиция живет на 3 из них.
>
> Если упадут 2 брокера (например, 2 дата-центра отключились), кластер останется жив, потому что у каждой партиции как
> минимум одна реплика (в живом брокере) остается.

---

## ISR (In-Sync Replicas)

> Не все фолловеры одинаково полезны. `ISR` - это список реплик, которые синхронизированы с лидером (не отстают более
> чем на заданное время или количество сообщений).
>
> Если фолловер отстал, завис или упал, он исключается из ISR.
>
> При записи с `acks=all` `Kafka` ждёт подтверждения от всех реплик в `ISR`, прежде чем считать запись успешной.
>
> Только реплики из ISR могут стать новым лидером.

### Если лидер умирает, происходит перевыборы:

- Контроллер (один из брокеров, следящий за кластером) замечает потерю лидера.
- Он берет список `ISR` для этой партиции.
- Выбирает самую "свежую" реплику из `ISR` и назначает ее новым лидером.
- Все продюсеры и консьюмеры узнают о новом лидере через метаданные.

### Минусы репликации:

- Если установить ёreplication.factor=3ё, данные хранятся в 3 экземплярах. Это требует в 3 раза больше дискового
  пространства.
- Если продюсер требует ёacks=allё, а один фолловер из `ISR` тормозит, запись замедлится (или уйдет в таймаут).

# Ключевые компоненты Kafka</b>

---

## Broker — сервер Kafka

- `Broker` — это отдельный узел (процесс) в кластере `Kafka`.
- Каждый `broker` имеет уникальный числовой идентификатор (`broker.id`).
- Хранит подмножество партиций всех топиков (как `leader`-ов, так и `follower`-ов).
- Обрабатывает запросы от `producer`-ов и `consumer`-ов.
- Не зависит от других брокеров напрямую — координация осуществляется через метаданные (ранее `ZooKeeper`, теперь
  `KRaft`).

Как добавляется новый брокер если продуктивности перестало хватать:

- Ты поднимаешь новый сервер с Kafka (broker4).
- Он стучится в кластер, говорит "я тут".
- Контроллер кластера перераспределяет партиции (ребалансировка), перемещая часть данных на новый брокер.
- Клиенты через запрос метаданных узнают об изменениях и начинают ходить на новый адрес.

> Ты работаешь с кластером как с единым целым через абстракцию "топик". Брокеров может быть 3 или 30 — твой код не
> меняется. Меняется только список `bootstrap.servers` в конфиге.

---

## Cluster — кластер брокеров

- Kafka работает как **распределённый кластер** из нескольких брокеров.
- Кластер обеспечивает:
    - масштабируемость (добавление брокеров увеличивает пропускную способность),
    - отказоустойчивость (репликация между брокерами),
    - балансировку нагрузки.
- Клиенты (`producer` / `consumer`) подключаются к **любому брокеру** — он перенаправляет запросы к нужному `leader`-у.

---

## `ZooKeeper` vs `KRaft` (Kafka Raft Metadata mode)

### `ZooKeeper` (до Kafka 3.3)

- До версии 3.3 Kafka полагалась на `Apache ZooKeeper` для хранения:
    - метаданных кластера (топики, партиции, ISR),
    - информации о состоянии брокеров,
    - координации выбора контроллера.
- Это добавляло сложность: нужно было управлять двумя системами.

### `KRaft` (рекомендуемый с Kafka 3.3+)

- Kafka теперь может работать **без** `ZooKeeper`, используя собственный протокол консенсуса на основе `Raft`.
- Метаданные хранятся **внутри самого Kafka-кластера**.
- Упрощает развертывание операционное управление кластером.
- Включается через параметр `process.roles=broker,controller` и указание `controller.quorum.voters`.

---

## `Controller` — координация кластера

- Один из брокеров в кластере назначается `Controller`-ом.
- Отвечает за:
    - управление партициями и репликами,
    - обработку изменений топологий (создание/удаление топиков),
    - инициацию `failover` при выходе брокера из строя.
- В `KRaft` режиме контроллеры образуют `quorum` (кворум) и достигают консенсуса через `Raft`.

---

## Log segments — хранение данных на диске

- Данные в каждой партиции хранятся как **лог** (`log`), разделённый на **сегменты** (`segments`).
- Каждый сегмент — файл фиксированного размера (по умолчанию 1 ГБ).
- Пример структуры на диске:

```text
/var/lib/kafka/data/
└── orders-0/
   ├── 00000000000000000000.log ← сегмент данных
   ├── 00000000000000000000.index ← индекс по offset
   ├── 00000000000000001234.log
   └── 00000000000000001234.index
```

- Kafka читает данные **последовательно**, что делает работу с диском очень эффективной.
- Старые сегменты удаляются по истечении **retention period** или при достижении лимита размера.

> Эти компоненты вместе формируют надёжную, распределённую систему потоковой передачи событий, способную обрабатывать
> огромные объёмы данных с минимальными задержками.

# Основные концепции и термины

---

## Topic (Топик)

> **Topic (Топик) - Логический канал**, в который продюсеры отправляют сообщения, а консьюмеры читают их.
> Аналог — таблица в БД или очередь в RabbitMQ, но с принципиальными отличиями.

**Именование:** отражает тип событий (например, `orders`, `user-payments`, `clickstream`).

**Создание:**

- Явное (через админку/CLI)
- Автоматическое (при отправке/подписке, если `auto.create.topics.enable=true`). В проде лучше отключать, чтобы избежать
  опечаток и неконтролируемых настроек.

**Ключевые характеристики:**

1. **Состоит из партиций.** Минимум одна. Чем больше партиций — тем выше параллелизм.
2. **Мульти-подписка.** Один топик могут читать тысячи независимых консьюмеров (разные consumer-группы) без влияния друг
   на друга.
3. **Хранение (`retention`).** В отличие от классических очередей, Kafka не удаляет сообщения сразу после прочтения. Они
   хранятся заданное время (по умолчанию 7 дней) или до достижения лимита размера. Консьюмер сам управляет своей
   позицией (`offset`).
4. **Неизменяемость (`immutable`).** Сообщения в топике нельзя изменить после записи.

**Физически:** топик — это просто имя в метаданных. Реальные данные живут в директориях на дисках брокеров, разбитые по
партициям и сегментам.

---

## Partition

## Partition (Партиция)

> Partition (Партиция) - Упорядоченная, неизменяемая последовательность сообщений с возрастающими `offset`-ами.

**Единица параллелизма и упорядоченности в Kafka.** Если топик — это логическая "река данных", то партиция — это
отдельный, строго упорядоченный ручей.

**Ключевая идея:** Партиция — это распределенный лог (журнал).

**Характеристики:**

1. **Offset как индекс.** Каждое вновь поступившее сообщение получает монотонно возрастающий номер (`offset`),
   уникальный в пределах партиции. Это позволяет консьюмеру точно сказать: "я прочитал всё до offset=5, давай
   следующее".

2. **Гарантия порядка (ключевое свойство).** Внутри **одной** партиции порядок сообщений **строго гарантирован** (в том
   порядке, в котором их прислал продюсер, если не было ретраев). Если нужен глобальный порядок для всех сообщений
   топика — можно использовать 1 партицию, но это убивает параллелизм.

3. **Физическое устройство.** Партиция — это не "таблица" в памяти, а набор файлов-сегментов на диске. Сообщения
   дописываются в конец активного сегмента (`append-only`), что обеспечивает высокую скорость записи.

4. **Распределение.** Партиции одного топика могут находиться на разных брокерах кластера. Это позволяет обрабатывать
   объем данных, который не влезает на один сервер.

5. **ISR (In-Sync Replicas).** Для отказоустойчивости у партиции есть реплики (`leader + followers`). Все операции
   записи/чтения идут через лидера, фолловеры синхронизируются асинхронно.

6. **Неизменяемость.** Сообщения в партиции никогда не изменяются после записи. Удаляются только при истечении срока
   хранения (`retention`) или при компакшене.

> Консьюмер, читающий партицию, гарантированно получает сообщения в том же порядке, в котором они
> были записаны. При этом разные консьюмеры в группе читают разные партиции (независимо), поэтому глобальный порядок по
> всему топику не гарантируется.

Партиции обеспечивают:

- параллелизм (разные потребители читают разные партиции).
- масштабируемость (данные распределяются по брокерам).
- упорядоченность **внутри одной партиции**.

---

## Offset (Смещение)

> 64-битное целое число, уникально идентифицирующее позицию сообщения внутри партиции.
>
> Offset начинается с `0` и монотонно возрастает.
>
> `Consumer` отслеживает текущий `offset`, чтобы продолжать чтение с нужного места.

**Технические детали:**

1. **Монотонность.** `Offset` начинается с 0 и увеличивается на 1 для каждого нового сообщения в партиции. Пропусков в
   нумерации нет (если сообщение записано, оно получило свой номер).

2. **Идентичность.** Идентифицировать сообщение в Kafka можно только тройкой: `[topic, partition, offset]`. Без указания
   партиции `offset` не имеет смысла.

3. **Управление позицией потребителя.** `Consumer` читает сообщения, передавая брокеру свой текущий `offset`. После
   обработки он **коммитит** (сохраняет) новый `offset`, сигнализируя: "я прочитал всё до `offset N`".

4. **Хранение `offset`-ов.** Раньше `offset`-ы хранились в `ZooKeeper`, сейчас — в специальном внутреннем топике
   `__consumer_offsets`. Kafka ведет его как обычный топик, что обеспечивает масштабируемость и отказоустойчивость.

5. **Сброс `offset` (`rewind`).** Благодаря тому, что `offset` хранится отдельно, а сообщения никуда не деваются (до
   истечения `retention`), консьюмер может:
    - Перечитать данные заново (`rewind to beginning`)
    - Пропустить "зависшие" сообщения (`jump to end`)
    - Откатиться к конкретному моменту времени (`seek by timestamp`)

6. **`At-least-once` семантика.** Типичный сценарий: `получили сообщения -> обработали -> закоммитили offset`. Если
   упали между обработкой и коммитом, после рестарта прочитаем те же сообщения снова (получим дубликаты). Это нормально
   для Kafka, идемпотентность обработки — задача приложения.

7. **Конкурентность.** `Offset` хранится **для каждой комбинации** `consumer-group + топик + партиция`. Поэтому разные
   сервисы (разные группы) могут читать один топик независимо, со своими `offset`-ами.

---

## Consumer Group (Группа потребителей)

> Группа потребителей, которые совместно читают один топик.
> - Каждая **партиция** назначается **ровно одному** потребителю в группе.
> - Это позволяет масштабировать обработку: больше потребителей — выше пропускная способность.
> - Потребители **в разных группах** читают одни и те же сообщения независимо.

**Механизм масштабирования и координации консьюмеров.** Это логическая единица, объединяющая одного или нескольких
потребителей для совместного чтения данных из топиков.

**Принцип работы:**

1. **Разделение партиций.** Каждая партиция в топике назначается **ровно одному** потребителю внутри группы. Два
   потребителя в одной группе **никогда не получат** одну и ту же партицию одновременно.
    - Это гарантирует, что каждое сообщение будет обработано только одним потребителем в группе (исключая повторные
      чтения при сбоях).

2. **Масштабирование горизонтально:**
    - Если потребителей **меньше**, чем партиций → часть потребителей получает несколько партиций.
    - Если потребителей **больше**, чем партиций → лишние потребители будут простаивать (не получат назначений).
    - Идеальное соотношение: кол-во потребителей = кол-во партиций.

3. **`Rebalance` (перебалансировка).** Когда потребитель присоединяется к группе, покидает её или падает, группа
   выполняет перебалансировку:
    - Останавливается чтение.
    - Партиции перераспределяются между живыми потребителями.
    - Чтение возобновляется с последних закоммиченных `offset`-ов.

4. **Независимость групп.** Разные consumer-группы читают **один и тот же топик независимо**.
    - Группа "email-senders" и группа "analytics-processors" могут параллельно читать `orders-topic`.
    - Каждая группа хранит свой собственный offset.
    - Это реализует паттерн "публикация-подписка" (pub/sub).

5. **Хранение состояния.** Kafka хранит `offset` только для коммитов от групп. Если группа остановлена на неделю, а
   `retention` топика — 7 дней, после рестарта она не найдет свои `offset` и начнет чтение с самого старого доступного
   сообщения (или согласно настройке `auto.offset.reset`).

**Пример для понимания:**

- Топик: `user-actions`, 6 партиций.
- Группа A: 2 потребителя → потребитель 1 читает партиции [0,1,2], потребитель 2 читает [3,4,5].
- Группа B: 6 потребителей → каждый читает ровно 1 партицию (максимальный параллелизм).
- Группа C: 1 потребитель → читает все 6 партиций последовательно.

## Commit (Коммит offset-а)

> Процесс сохранения текущего `offset`-а как «успешно обработанного».

- **Auto-commit**: Kafka периодически (`auto.commit.interval.ms`) сохраняет offset автоматически.
- **Manual commit**: приложение явно вызывает `commitSync()` или `commitAsync()` после обработки.

### Commit (Коммит offset'а)

> **Commit (Коммит offset'а) - Механизм фиксации позиции потребителя.**
>
> Коммит — это сигнал Kafka о том, что сообщения до определенного offset'а
> обработаны и при сбое можно начинать чтение со следующего.

- Без коммитов после перезапуска консьюмер каждый раз читал бы топик с самого начала (offset=0).
- При коммите консьюмер отправляет специальное сообщение во внутренний топик `__consumer_offsets` с ключом
  `[groupId, topic, partition]` и значением `offset`.

### Auto-commit (Автоматический коммит)

> - Консьюмер раз в N миллисекунд (`auto.commit.interval.ms` = 5000 по умолчанию) коммитит максимальный offset среди уже
    полученных сообщений.
> - Коммит происходит **синхронно** в фоновом потоке.
>
> - Параметр `enable.auto.commit=true` (по умолчанию).

**Риски:**

1. **Повторная обработка.** Получили пачку сообщений (offsets 100-110). Обработали 105-е, но не дошли до 110-го. Через 5
   секунд авто-коммит сохранит offset = 111 (если poll вернул сообщения до 110). Если консьюмер упадет после обработки
   105-го, но до коммита, после рестарта он начнет с 100-го — сообщения 100-105 будут обработаны повторно.
2. **"Мертвые души".** Если консьюмер "завис" на обработке сообщения 105-го, а таймер авто-коммита сработал — offset
   уедет вперед, а сообщение 105-е так и не обработано (потеря данных).

**Где использовать:** Только там, где допустима потеря данных или дубликаты не критичны (логи, метрики).

### Manual commit (Ручной коммит)

> `enable.auto.commit=false` - приложение само вызывает `commitSync()` или `commitAsync()`.

**Типичные стратегии:**

### Коммит после обработки каждого сообщения

```java
while(true) {
    ConsumerRecords<String, String> records = consumer.poll(100);
    for(ConsumerRecord record :records) {
        process(record);            // обработали
        consumer.commitSync();      // закоммитили offset этого сообщения
    }
}
```

- ✔️ Минимизирует повторные обработки.
- ❌ Тормозит (RTT на каждое сообщение).

### Коммит после пачки (правильный вариант)

```java
while(true){
ConsumerRecords<String, String> records = consumer.poll(100);
processBatch(records);          // обработали всю пачку
consumer.commitSync();          // закоммитили последний offset пачки
}
```

- ✔️ Баланс производительности и надежности.
- ⚠️ При сбое после `processBatch`, но до `commitSync` — вся пачка уйдет на повтор.

### Асинхронный коммит с колбэком

```java
consumer.commitAsync((offsets, exception) ->{
        if(exception !=null)log.error("Commit failed for {}",offsets);
});
```

✔️ Не блокирует поток.
⚠️ Нужен повтор при ошибке (retry) + финальный синхронный коммит при graceful shutdown.

---

## Идемпотентность и exactly-once

Ручной коммит сам по себе дает только `at-least-once` (минимум один раз). Чтобы получить `exactly-once` (ровно один
раз), нужно:

- Сохранять результат обработки и offset в одну транзакцию (например, в БД).
- Использовать Kafka Transactions API (продвинутый уровень).

---

## Retention policy

Правило хранения сообщений в топике. По умолчанию — **время** (`retention.ms`, обычно 7 дней).  
Альтернатива — **размер** (`retention.bytes`).  
Сообщения удаляются независимо от того, были ли они прочитаны.

---

## Key / Value сериализация

Каждое сообщение состоит из:

- **Key** (опционально): используется для определения партиции.
- **Value**: полезная нагрузка.
- Оба поля — байтовые массивы; их преобразование в/из объектов выполняется через **сериализаторы**.

Пример:

Key = `"user-123"` → всегда попадает в одну и ту же партицию → гарантирует порядок событий по пользователю.

> **Kafka не знает о типах данных.** Она хранит и передает сообщения исключительно как **массивы байтов** (`byte[]`). За
> преобразование объектов Java в байты и обратно отвечают сериализаторы и десериализаторы.

### Разница между Key и Value:

| Аспект              | Key                                                                        | Value                      |
|---------------------|----------------------------------------------------------------------------|----------------------------|
| Обязательность      | ❌ Опционален                                                               | ✅ Всегда есть              |
| Назначение          | Определяет партицию (партиционирование)                                    | Содержит бизнес-данные     |
| Типичное содержимое | ID сущности (user-123, order-456)                                          | JSON, Avro, Protobuf       |
| Влияние на порядок  | Сообщения с одинаковым key попадают в одну партицию → порядок гарантирован | Не влияет на маршрутизацию |

### Как работает сериализация

**Producer (пример с Spring Kafka):**

- Конфигурация

```java
props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, JsonSerializer.class);
```

- Отправка

```java
producer.send(new ProducerRecord<>("topic", "user-123",new Order(...)));
// Key "user-123" → byte[] через StringSerializer
// Value Order объект → byte[] через JsonSerializer (в JSON)
```

- Consumer:

```java
props.put(ConsumerConfig.KEY_DESERIALIZER_CLASS_CONFIG, StringDeserializer.class);
props.put(ConsumerConfig.VALUE_DESERIALIZER_CLASS_CONFIG, JsonDeserializer.class);

ConsumerRecords<String, Order> records = consumer.poll(...);
// Kafka сама соберет объекты обратно
```

---

### Популярные сериализаторы

| Тип данных | Сериализатор                          | Примечание                                      |
|------------|---------------------------------------|-------------------------------------------------|
| Строки     | `StringSerializer`                    | Просто, но нет схемы                            |
| JSON       | `JsonSerializer` (Spring) / `Jackson` | Удобно для dev, но ломается при изменении полей |
| Avro       | `KafkaAvroSerializer` (Confluent)     | Со схемой в Schema Registry, эволюция данных    |
| Protobuf   | `ProtobufSerializer`                  | Строгая типизация, быстрее JSON                 |
| Byte array | `ByteArraySerializer`                 | Для сырых данных                                |

---

### Роль Key в партиционировании

Стандартный **partitioner** (если не задан свой) вычисляет партицию так:

`hash(key) % количество_партиций`

**Следствия:**

1. Все сообщения с `key = "user-123"` попадут в **одну и ту же** партицию.
2. Гарантируется порядок событий для этого пользователя.
3. Если key = `null` — распределение `round-robin` (равномерно, но порядок не гарантирован).

**Пример бизнес-логики:**

- Key = `customerId` → все действия клиента строго упорядочены.
- Key = `null` для массовых событий (клики, логи) — важна пропускная способность.

### Проблема эволюции данных

При смене версии объекта (добавили поле):

- JSON: старые консьюмеры упадут с ошибкой десериализации (нет поля).
- Avro + Schema Registry: консьюмеры узнают о совместимости схем и смогут читать и старые, и новые сообщения.

**Для продакшена** рекомендуется Avro/Protobuf + Schema Registry, чтобы избежать "проклятия JSON".


### Резюме

1. Key определяет **партицию** и **порядок**.
2. Value содержит **данные**.
3. Сериализаторы превращают объекты в байты и обратно.
4. Выбор сериализатора влияет на совместимость при изменениях схемы данных.

## Message timestamp

Каждое сообщение содержит временную метку:

- **CreateTime** (по умолчанию): время отправки `producer`-ом.
- **LogAppendTime**: время записи на диск `broker`-ом (если включено на стороне брокера).

Используется для:

- потоковой обработки по времени.
- `retention` на основе времени.
- отладки.

---

## Log compaction

Особый режим хранения, при котором Kafka сохраняет **только последнее значение для каждого ключа**.  
Полезно для:

- хранения состояний (например, профиль пользователя),
- реализации `changelog`-ов.

Включается через `cleanup.policy=compact`.

---

## Exactly-once semantics (`EOS`)

Гарантия, что каждое сообщение будет обработано **ровно один раз**, даже при сбоях.  
Достигается через:

- **Idempotent Producer** (гарантия уникальности записи на стороне брокера),
- **Transactional Producer** (атомарная отправка в несколько партиций),
- **Consumer offset commit в той же транзакции**, что и запись результата.

Требует настройки `enable.idempotence=true` и/или использования транзакций.

---

## Idempotent Producer

Producer, который гарантирует, что повторные отправки (из-за сетевых ошибок) **не создадут дубликатов**.  
Включается через `enable.idempotence=true`.  
Kafka использует внутренний `producer.id` и `sequence number` для дедупликации.

---

## Transactional Producer (Транзакционный продюсер)

> **Механизм атомарной записи в несколько партиций.** Позволяет отправить набор сообщений так, что либо все они будут записаны в топики, 
либо ни одно (всё или ничего).

### Зачем нужны транзакции в Kafka?

В распределенной системе без транзакций возможны сценарии частичной записи:

**Проблема:**

- Сервис обрабатывает событие и должен обновить БД + отправить сообщения в 2 разных топика.
- Отправили в топик A, но при отправке в топик B произошла ошибка.
- Состояние системы неконсистентно: данные в БД обновлены, но второе сообщение потеряно.

**Решение:** транзакционный продюсер гарантирует, что все отправленные в рамках транзакции сообщения будут применены атомарно.

### Как это работает

**Принцип:**

1. Продюсер инициирует транзакцию (`beginTransaction()`)
2. Отправляет сообщения в любые партиции (даже разных топиков)
3. Фиксирует транзакцию (`commitTransaction()`) — все сообщения становятся видны консьюмерам
4. При ошибке — откат (`abortTransaction()`) — сообщения не доставляются

**Пример кода:**

```java
// Инициализация транзакционного продюсера
props.put(ProducerConfig.TRANSACTIONAL_ID_CONFIG, "prod-1");
KafkaProducer<String, String> producer = new KafkaProducer<>(props);
producer.initTransactions();

// Использование
try {
    producer.beginTransaction();

    producer.send(new ProducerRecord<>("topic-a", "key1", "value1"));
    producer.send(new ProducerRecord<>("topic-b", "key2", "value2"));
    
    producer.commitTransaction();
} catch (Exception e) {
    producer.abortTransaction();
}
```

### Важные детали

**1. TransactionalId**

Уникальный идентификатор транзакционного продюсера. Kafka использует его для:

- Гарантии идемпотентности (при повторах не создаются дубли)
- Восстановления незавершенных транзакций (если продюсер упал, транзакция будет отменена)

**2. Чтение транзакционных данных**

Консьюмер должен быть настроен на уровень изоляции:

- `isolation.level=read_committed` — видит только закоммиченные транзакции (пропускает незавершенные и отмененные)
- `isolation.level=read_uncommitted` — видит всё (по умолчанию)

**3. Consumer транзакции**

Kafka не поддерживает распределенные транзакции "чтение-обработка-запись" из коробки. 
Паттерн `consume-transform-produce` реализуется через:

```java
// Читаем из одного топика, пишем в другой в одной транзакции
consumer.poll(); // читаем
producer.beginTransaction();
producer.send(...); // пишем результат
producer.sendOffsetsToTransaction(...); // коммитим offset прочитанных сообщений в ту же транзакцию
producer.commitTransaction();
```


### Exactly-once семантика (EOS)

Транзакции + идемпотентный продюсер (`enable.idempotence=true`) дают exactly-once:

1. **Идемпотентность** — защита от дублей при повторных отправках (Producer ID + Sequence Number)
2. **Транзакции** — атомарность записи в несколько партиций + атомарный коммит offset'ов

---

### Оверхед и ограничения

⚠️ Транзакции добавляют накладные расходы:
- Дополнительные RTT (round-trip) для координации
- Хранение состояния транзакций в специальном топике `__transaction_state`
- Чуть выше латентность

По умолчанию `transaction.timeout.ms = 60000` (1 минута). Если транзакция выполняется дольше — будет прервана.

---

### Когда использовать

- Операции, требующие атомарности записи в несколько топиков
- Exactly-once processing (читаем из Kafka → обрабатываем → пишем в Kafka)
- Финансовые транзакции, инвентари, заказы (где важна консистентность)

### Резюме

| Без транзакций | С транзакциями |
|----------------|----------------|
| Возможна частичная запись | Атомарность "всё или ничего" |
| At-least-once минимум | Exactly-once (с идемпотентностью) |
| Нет координации между партициями | Координация через Transaction Coordinator |
| Выше производительность | Ниже производительность, но гарантии |

# Модель потребления: Consumer Groups и Rebalancing

## Как работает Consumer Group

> Consumer Group — это логическая группа потребителей, совместно читающих один или несколько топиков.  
>
> Kafka гарантирует, что **каждая партиция назначается ровно одному потребителю** внутри группы.  

Это позволяет:

- масштабировать обработку: больше потребителей → выше параллелизм,
- обеспечивать отказоустойчивость: при падении одного потребителя его партиции перераспределяются.

Пример:  
Топик `events` с 6 партициями:
- 1 потребитель → читает все 6 партиций,
- 3 потребителя → каждый получает по 2 партиции,
- 8 потребителей → 6 активны, 2 простаивают («лишние»).

> ⚠️ Количество активных потребителей в группе не может превышать количество партиций.

## Partition assignment стратегии

При старте или изменении состава группы Kafka выполняет **назначение партиций** (`partition assignment`).  
Стратегия определяется параметром `partition.assignment.strategy`. 

Основные варианты:

- RangeAssignor (по умолчанию до Kafka 2.4)
  - Сортирует партиции и потребителей по имени.
  - Назначает непрерывные диапазоны партиций.
  - Может привести к **неравномерной нагрузке**, если несколько топиков читаются одной группой.

- RoundRobinAssignor
  - Равномерно распределяет партиции по кругу.
  - Более сбалансирован, но требует одинакового набора подписок у всех потребителей.

- CooperativeStickyAssignor (рекомендуется с Kafka 2.4+)
  - Минимизирует перемещение партиций между rebalance-ами.
  - Поддерживает **кооперативный** режим: потребители могут отдавать партиции без полной остановки.
  - Позволяет избежать «сторма» rebalance при частых изменениях.

> 💡 В Spring Boot используется `CooperativeStickyAssignor` по умолчанию (если указана версия Kafka ≥ 2.4).

## Rebalancing: причины, последствия, как минимизировать

> **Rebalancing** — процесс перераспределения партиций между потребителями в группе.

### Причины rebalance:

- Запуск нового потребителя в группе.
- Остановка или падение существующего потребителя.
- Изменение подписки на топики (`.subscribe()`).
- Превышение `max.poll.interval.ms` (потребитель слишком долго обрабатывает batch).
- Превышение `session.timeout.ms` (нет heartbeat-ов от потребителя).

### Последствия:

- Все потребители **останавливают обработку** на время rebalance.
- Возможна **повторная обработка** сообщений (если offset не был закоммичен).
- Частые rebalance-ы снижают пропускную способность и увеличивают задержки.

### Как минимизировать:

- Увеличьте `session.timeout.ms` (например, до 30 сек) и `heartbeat.interval.ms` (в 3 раза меньше session timeout).
- Уменьшите объём данных за один `poll()` или увеличьте `max.poll.interval.ms`, если обработка долгая.
- Избегайте блокирующих операций в consumer thread.
- Используйте `CooperativeStickyAssignor` для плавного перераспределения.

## Manual vs Automatic offset commit

### Automatic commit (`enable.auto.commit=true`)

- Offset коммитится каждые `auto.commit.interval.ms` (по умолчанию 5 сек).
- Простота, но риск **потери данных**: если consumer упадёт после обработки, но до коммита — сообщение будет повторно обработано.
- Или **пропуска данных**: если коммит произойдёт до обработки (при неосторожном использовании).

### Manual commit

- Явный вызов `consumer.commitSync()` (блокирующий) или `commitAsync()` (неблокирующий).
- Позволяет коммитить **только после успешной обработки** → гарантия **at-least-once**.
- Для **exactly-once** требуется транзакционный коммит вместе с записью результата.

Рекомендация: **всегда используйте ручной коммит в production**, если не готовы к потере или дублированию данных.

## Потребление из нескольких топиков

Consumer может подписаться на несколько топиков:

```java
consumer.subscribe(Arrays.asList("topic-A", "topic-B"));
```

- Все партиции из всех топиков объединяются в единый пул.
- Назначение партиций происходит **единообразно** для всей группы.
- Обработка сообщений из разных топиков происходит в одном цикле `poll()`.

Альтернатива — использовать `assign()` для ручного указания конкретных партиций (без consumer group), 
но тогда теряются преимущества автоматического балансирования.

# Установка и запуск Kafka локально

## Скачивание Kafka

Официальный дистрибутив Apache Kafka доступен на [https://kafka.apache.org/downloads](https://kafka.apache.org/downloads).  
Для локальной разработки подойдёт **Binary downloads** (например, `kafka_2.13-3.7.0.tgz`).

Распакуйте архив:

```text
tar -xzf kafka_2.13-3.7.0.tgz
cd kafka_2.13-3.7.0
```

> 💡 Версия `2.13` — это версия Scala, на которой скомпилирован Kafka. Для пользователя это не имеет значения.

## Запуск в режиме KRaft (без ZooKeeper)

Начиная с Kafka 3.3+, рекомендуется использовать **KRaft mode** (Kafka Raft Metadata mode).

### Шаг 1: Сгенерируйте cluster ID

```bash
bin/kafka-storage.sh random-uuid
# Пример вывода: 4cGhJ7vTR9qXnYzWmLpQrA
```

Сохраните этот UUID — он понадобится дальше.

### Шаг 2: Отформатируйте storage directory

Замените `<UUID>` на сгенерированный выше:

```bash
bin/kafka-storage.sh format -t <UUID> -c config/kraft/server.properties
```

### Шаг 3: Запустите Kafka сервер

```bash
bin/kafka-server-start.sh config/kraft/server.properties
```

По умолчанию:
- Kafka слушает порт `9092`,
- метаданные хранятся в `/tmp/kraft-combined-logs`.

> Теперь Kafka запущена без ZooKeeper.

## Создание топика через CLI

Создадим топик `demo-topic` с 3 партициями:

```bash
bin/kafka-topics.sh --create \
--topic demo-topic \
--partitions 3 \
--replication-factor 1 \
--bootstrap-server localhost:9092
```

> В single-node кластере `replication-factor` должен быть `1`.

Просмотр списка топиков:

```bash
bin/kafka-topics.sh --list --bootstrap-server localhost:9092
```

Описание топика:

```bash
bin/kafka-topics.sh --describe --topic demo-topic --bootstrap-server localhost:9092
```

## Отправка сообщений через консольный producer

```bash
bin/kafka-console-producer.sh \
--topic demo-topic \
--bootstrap-server localhost:9092
```

Вводите сообщения по одному (каждая строка — отдельное сообщение):

```bash
>Hello, Kafka!
>User logged in
>Order created
```

Чтобы выйти — `Ctrl+D` (Unix) или `Ctrl+Z + Enter` (Windows).

## Получение сообщений через консольный consumer

Запустите consumer в новом терминале:

```bash
bin/kafka-console-consumer.sh \
--topic demo-topic \
--bootstrap-server localhost:9092 \
--from-beginning
```

Флаг `--from-beginning` означает чтение с самого первого сообщения (offset = 0).  
Без него consumer читает только новые сообщения (с текущего конца лога).

> 💡 Consumer автоматически создаёт consumer group с случайным именем. Для явного указания используйте `--group my-group`.

## Полезные замечания

- Все данные по умолчанию хранятся в `/tmp`, поэтому **удаляются при перезагрузке**.
- Для постоянного хранения измените `log.dirs` в `config/kraft/server.properties`.
- Порт `9092` можно изменить через `listeners=PLAINTEXT://:9093`.

Теперь у вас есть полностью рабочий локальный Kafka-кластер в KRaft-режиме — готовый к интеграции с Java и Spring Boot.

# Kafka Clients: Producer и Consumer API (на чистой Java)

## Настройка Properties для Producer

Producer настраивается через `java.util.Properties`:

```java
Properties props = new Properties();
props.put("bootstrap.servers", "localhost:9092");
props.put("key.serializer", "org.apache.kafka.common.serialization.StringSerializer");
props.put("value.serializer", "org.apache.kafka.common.serialization.StringSerializer");
props.put("acks", "all");                          // ждать подтверждения от всех ISR
props.put("retries", 3);                           // повтор при временных ошибках
props.put("enable.idempotence", true);             // включить idempotent producer

KafkaProducer<String, String> producer = new KafkaProducer<>(props);
```

## Пример отправки сообщения

```java
ProducerRecord<String, String> record = new ProducerRecord<>(
    "demo-topic",      // топик
    "user-123",        // ключ (опционально)
    "User logged in"   // значение
);
```

```java
// Синхронная отправка (блокирующая)
try {
    RecordMetadata metadata = producer.send(record).get();
    System.out.printf("Sent to partition %d, offset %d%n", metadata.partition(), metadata.offset());
} catch (Exception e) {
    e.printStackTrace();
}
```

```java
// Асинхронная отправка с Callback
producer.send(record, (metadata, exception) -> {
    if (exception != null) {
        System.err.println("Send failed: " + exception.getMessage());
    } else {
        System.out.printf("Async sent to partition %d%n", metadata.partition());
    }
});
```

> ⚠️ Не забывайте вызывать `producer.close()` при завершении работы.

## Настройка Properties для Consumer

```java
Properties props = new Properties();
props.put("bootstrap.servers", "localhost:9092");
props.put("group.id", "my-consumer-group");
props.put("key.deserializer", "org.apache.kafka.common.serialization.StringDeserializer");
props.put("value.deserializer", "org.apache.kafka.common.serialization.StringDeserializer");
props.put("auto.offset.reset", "earliest");        // если нет коммита — читать с начала
props.put("enable.auto.commit", false);            // ручной коммит (рекомендуется)

KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);
```

## Пример чтения сообщений

```java
consumer.subscribe(Arrays.asList("demo-topic"));

try {
    while (true) {
        ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
        
        for (ConsumerRecord<String, String> record : records) {
            System.out.printf("Received: key=%s, value=%s, partition=%d, offset=%d%n", 
                              record.key(), record.value(), record.partition(), record.offset());

            // Обработка сообщения...
        }

        // Ручной коммит после обработки всего batch’а
        consumer.commitSync();
    }
} catch (WakeupException e) {
        // Это НЕ ошибка! Это сигнал к завершению.
        // Мы просто логируем и выходим из цикла
        System.out.println("Получен сигнал wakeup, завершаем работу");
} finally {
        // ВАЖНО: здесь мы закрываем consumer и коммитим последние offset'ы
        consumer.close();
        System.out.println("Consumer закрыт");
}
```

## Асинхронная отправка и Callback

Callback позволяет реагировать на успех/ошибку без блокировки основного потока.  
Полезно для высоконагруженных producer’ов.

## Обработка исключений и retries

- **Retriable exceptions** (например, `NetworkException`, `NotLeaderForPartitionException`) — можно повторять.
- **Non-retriable exceptions** (например, `SerializationException`) — требуют исправления кода.

Idempotent producer (`enable.idempotence=true`) автоматически обрабатывает повторные отправки без дубликатов.

## Close и graceful shutdown

Всегда закрывайте клиенты:

- `producer.close(Duration.ofSeconds(10))` — дожидается отправки всех сообщений в буфере.
- `consumer.wakeup()` — прерывает текущий `poll()`, чтобы выйти из цикла.

Пример graceful shutdown:

```java
Runtime.getRuntime().addShutdownHook(new Thread(() -> {
    consumer.wakeup(); // прерывает poll()
}));
```

Эти базовые примеры показывают, как работать с Kafka без Spring. Они полезны для понимания низкоуровневого поведения, 
которое Spring абстрагирует.

# Сериализация и десериализация

## Встроенные сериализаторы

Kafka работает с байтовыми массивами, но предоставляет встроенные сериализаторы для простых типов:

- `StringSerializer` / `StringDeserializer`
- `ByteArraySerializer` / `ByteArrayDeserializer`
- `IntegerSerializer`, `LongSerializer` и др.

Пример конфигурации:

```java
props.put("key.serializer", "org.apache.kafka.common.serialization.StringSerializer");
props.put("value.serializer", "org.apache.kafka.common.serialization.StringSerializer");
```

⚠️ Сериализаторы указываются как **строки с полным именем класса**.

## Использование JSON (Jackson)

Для передачи объектов часто используется JSON. Создадим кастомный сериализатор:

```java
public class JsonSerializer<T> implements Serializer<T> {

    private final ObjectMapper objectMapper = new ObjectMapper();

    @Override
    public byte[] serialize(String topic, T data) {
        try {
            return objectMapper.writeValueAsBytes(data);
        } catch (Exception e) {
            throw new SerializationException("Error serializing JSON", e);
        }
    }
}
```

Аналогично — десериализатор:

```java
public class JsonDeserializer<T> implements Deserializer<T> {

    private final ObjectMapper objectMapper = new ObjectMapper();
    private final Class<T> targetType;

    public JsonDeserializer(Class<T> targetType) {
        this.targetType = targetType;
    }

    @Override
    public T deserialize(String topic, byte[] data) {
        try {
            return objectMapper.readValue(data, targetType);
        } catch (Exception e) {
            throw new SerializationException("Error deserializing JSON", e);
        }
    }
}
```

Использование в producer:

```java
props.put("value.serializer", "com.example.JsonSerializer");

// Передаём объект напрямую
producer.send(new ProducerRecord<>("orders", order));
```

В consumer:

```java
props.put("value.deserializer", "com.example.JsonDeserializer");
// Но так не сработает — нужно передать тип
```

Лучше использовать фабрику или обёртку. В Spring Boot это решается проще.

## Avro и Schema Registry

- `Avro` — компактный бинарный формат со строгой схемой
- `Schema Registry` — внешний сервис, хранящий схемы и управляющий их версиями
- Вместе дают:
  - Экономию места (до 80% по сравнению с JSON)
  - Безопасную эволюцию данных (без остановки сервисов)
  - Совместимость между командами и сервисами

Для production это стандарт де-факто. JSON оставляют для прототипов и внутренних сервисов, где схема не меняется годами.

### Avro

> **Avro** — это не просто сериализатор, а система сериализации данных с тремя ключевыми особенностями:

1. **Бинарный формат** — данные в Avro занимают мало места (компактнее JSON)
2. **Схема в JSON** — описание структуры данных хранится отдельно от самих данных
3. **Эволюция схем** — можно менять структуру данных, сохраняя совместимость

**Как выглядит Avro-схема:**

```json
{
  "type": "record",
  "name": "Order",
  "namespace": "com.example",
  "fields": [
    {"name": "id", "type": "string"},
    {"name": "amount", "type": "double"},
    {"name": "userId", "type": "string"},
    {"name": "createdAt", "type": "long"}
  ]
}
```

> Avro не хранит имена полей в каждом сообщении. Вместо этого он читает схему и знает: "первое поле — id (string), второе — amount (double)...". Это даёт компактность и скорость.

### Schema Registry (реестр схем)

Schema Registry — это отдельный сервис, который:

- Хранит все схемы топиков (по одной на каждую версию)
- Присваивает ID каждой версии схемы
- Проверяет совместимость при публикации новых версий
- Отдаёт схему по ID консьюмерам

Как работает:

- Продюсер перед отправкой сообщения: "Registry, вот моя схема (или её ID)".
- Registry проверяет совместимость с предыдущей версией, сохраняет и возвращает ID.
- Продюсер пишет в Kafka: `[ID схемы][бинарные данные Avro]`
- Консьюмер читает: берёт ID, идёт в Registry за схемой, десериализует данные.

### Пример полного цикла

Продюсер:

```java
// Конфигурация
props.put("value.serializer", "io.confluent.kafka.serializers.KafkaAvroSerializer");
props.put("schema.registry.url", "http://localhost:8081");

// Отправка (Order — сгенерированный по Avro-схеме класс)
Order order = Order.newBuilder()
    .setId("123")
    .setAmount(100.0)
    .setUserId("user-1")
    .setCreatedAt(System.currentTimeMillis())
    .build();

producer.send(new ProducerRecord<>("orders", order));
```

Консьюмер:

```java
props.put("value.deserializer", "io.confluent.kafka.serializers.KafkaAvroDeserializer");
props.put("schema.registry.url", "http://localhost:8081");

// Читаем
ConsumerRecords<String, Order> records = consumer.poll(100);
// Order автоматически десериализован с правильной схемой
```

### Эволюция схем (правила совместимости)

Schema Registry поддерживает разные стратегии:

- `BACKWARD` - Можно удалять поля, добавлять поля с default (старые данные читаются новым кодом)
- `FORWARD` - Можно добавлять поля (новые данные читаются старым кодом)
- `FULL` - И `BACKWARD`, и `FORWARD`
- `NONE` - Всё можно, но сломается

## Custom Serializer/Deserializer

Если вы не используете JSON или Avro, можно реализовать свои:

- Реализуйте интерфейсы `org.apache.kafka.common.serialization.Serializer<T>` и `Deserializer<T>`.
- Убедитесь, что они **потокобезопасны** (лучше без состояния).
- Обрабатывайте исключения — иначе consumer/producer упадёт.

Пример для простого POJO через `ObjectOutputStream` (не рекомендуется в production, но для понимания):

```java
public class ObjectSerializer implements Serializer<Object> {

    @Override
    public byte[] serialize(String topic, Object data) {
    try (ByteArrayOutputStream bos = new ByteArrayOutputStream(); 
         ObjectOutputStream out = new ObjectOutputStream(bos)) {
        out.writeObject(data);
        return bos.toByteArray();
    } catch (IOException e) {
        throw new SerializationException(e);
        }
    }
}
```

> ❌ Такой подход небезопасен, несовместим между языками и не поддерживает эволюцию схемы. Используйте только для экспериментов.

## Рекомендации

- **Избегайте Java-сериализации** (`ObjectSerializer`) — она не переносима и уязвима.
- **Предпочитайте текстовые форматы (JSON)** для простоты отладки на старте.
- **Переходите на Avro/Protobuf** при росте сложности и требований к совместимости.
- Всегда **логируйте ошибки сериализации** — они критичны для стабильности.

# Интеграция Kafka со Spring Framework</b>

## Обзор Spring for Apache Kafka

> Spring предоставляет высокоуровневую абстракцию над нативным Kafka Client через проект **Spring for Apache Kafka**.  
> 
> Он входит в экосистему Spring и тесно интегрируется с Spring Boot.

Основные возможности:

- Автоматическая конфигурация producer/consumer,
- Аннотации для потребления (`@KafkaListener`),
- Поддержка транзакций и exactly-once,
- Встроенная обработка ошибок,
- Тестовая поддержка (`@EmbeddedKafka`).

## Основные зависимости

В `pom.xml` (Maven):

```xml
<dependency>
    <groupId>org.springframework.kafka</groupId>
    <artifactId>spring-kafka</artifactId>
</dependency>
```

В `build.gradle` (Gradle):

```groovy
implementation 'org.springframework.kafka:spring-kafka'
```

Spring Boot автоматически настраивает бины при наличии этой зависимости.

## Конфигурация через application.yml

Пример минимальной конфигурации:

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer
      acks: all
      enable-idempotence: true
    consumer:
      group-id: my-spring-consumer
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      auto-offset-reset: earliest
      enable-auto-commit: false
```

> 💡 Spring Boot автоматически создаёт `ProducerFactory`, `ConsumerFactory`, `KafkaTemplate` и другие компоненты.

## ProducerFactory и KafkaTemplate

- `ProducerFactory` — фабрика для создания `KafkaProducer`.
- `KafkaTemplate` — высокоуровневый шаблон для отправки сообщений.

Spring Boot создаёт `KafkaTemplate<String, String>` автоматически при указании сериализаторов.

Пример внедрения:

```java
@Autowired
private KafkaTemplate<String, String> kafkaTemplate;
```

## ConsumerFactory и @KafkaListener

- `ConsumerFactory` — фабрика для `KafkaConsumer`.
- `@KafkaListener` — аннотация для методов, которые должны получать сообщения.

Пример слушателя:

```java
@KafkaListener(topics = "demo-topic", groupId = "my-group")
public void listen(String message) {
    System.out.println("Received: " + message);
}
```

Spring автоматически:

- создаёт consumer,
- подписывается на топик,
- вызывает метод при получении сообщения,
- управляет offset-ами (в зависимости от настроек).

## Гибкость конфигурации

Если нужны разные настройки для разных producer’ов или consumer’ов, можно объявить свои фабрики:

```java
@Bean
public ProducerFactory<String, MyEvent> myEventProducerFactory() {
    Map<String, Object> props = new HashMap<>();
    props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
    props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
    props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, JsonSerializer.class);
    return new DefaultKafkaProducerFactory<>(props);
}

@Bean
public KafkaTemplate<String, MyEvent> myEventKafkaTemplate() {
    return new KafkaTemplate<>(myEventProducerFactory());
}
```

Аналогично для consumer’ов.

## Поддержка типов

> Spring Kafka поддерживает автоматическую десериализацию в объекты, если указан соответствующий десериализатор 
(например, `JsonDeserializer` из `spring-kafka`):

```yaml
spring:
  kafka:
    consumer:
      value-deserializer: org.springframework.kafka.support.serializer.JsonDeserializer
      properties:
        spring.json.trusted.packages: com.example.model
```


Теперь можно писать:

```java
@KafkaListener(topics = "orders")
public void handleOrder(Order order) {
    // order — уже Java-объект
}
```

> ⚠️ Указание `spring.json.trusted.packages` обязательно по соображениям безопасности.

Эта интеграция значительно упрощает работу с Kafka в Spring-приложениях, сохраняя при этом гибкость для тонкой настройки.
