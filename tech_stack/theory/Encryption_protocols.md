# Алгоритмы шифрования трафика

## Симметричное шифрование (Symmetric Encryption)

> Один ключ для шифрования и расшифровки. Быстрое, подходит для шифрования больших объёмов данных.

- AES (Advanced Encryption Standard)
    - Блочный шифр (128-битные блоки), ключи 128/192/256 бит
    - Режимы: GCM (authenticated encryption), CBC (устаревающий)
    - Применение: TLS 1.2/1.3, шифрование дисков, VPN, защита данных в rest
- ChaCha20-Poly1305
    - Потоковый шифр + MAC для аутентификации
    - Эффективен на устройствах без аппаратного AES, устойчив к timing-атакам
    - Применение: TLS 1.3, мобильные приложения, WireGuard
- ⚠️ DES / 3DES — устарели, не использовать (малый размер ключа, уязвимости)

## Асимметричное шифрование (Asymmetric Encryption)

> Пара ключей: публичный (шифрование) и приватный (расшифровка). Медленнее симметричного, используется для обмена ключами
и подписей.

- RSA (Rivest–Shamir–Adleman)
    - Основан на сложности факторизации больших чисел
    - Ключи ≥2048 бит (рекомендуется 3072+), поддержка подписи и шифрования
    - Применение: TLS handshake, цифровые подписи, SSH
- ECC (Elliptic Curve Cryptography) / ECDSA / ECDH
    - Основан на алгебре эллиптических кривых; та же стойкость при меньшем размере ключа
    - Ключи 256 бит ≈ RSA 3072 бит; быстрее и экономичнее
    - Применение: TLS 1.3, мобильные/IoT-устройства, сертификаты Let's Encrypt

## Протоколы обмена ключами (Key Exchange)

> Когда стороны безопасно договариваются об общем симметричном ключе по незащищённому каналу.

- Diffie-Hellman (DH) / ECDH (Elliptic Curve DH)
    - Позволяет вычислить общий секрет, не передавая его по сети
    - ECDH — эллиптическая кривая версия: быстрее, короче ключи
    - Применение: TLS handshake, установление сессионных ключей
- Perfect Forward Secrecy (PFS)
    - Временные ключи (ephemeral, DHE/ECDHE) обеспечивают стойкость к компрометации долгосрочных ключей
    - Даже при утечке приватного ключа сервера прошлые сессии остаются защищёнными
    - Применение: современный TLS 1.3 (обязательно), безопасные мессенджеры

## Гибридные схемы и протоколы (Hybrid Encryption)

> Практически весь трафик шифруется гибридно: асимметрия для handshake + симметрия для данных.

- TLS 1.2 / TLS 1.3
    - Handshake: асимметричная криптография (RSA/ECDH) + аутентификация сертификатами
    - Data: симметричный шифр (AES-GCM / ChaCha20-Poly1305) + AEAD для целостности
    - TLS 1.3 убрал уязвимые алгоритмы, оставил только PFS-шифры
    - SSL и TLS 1.0/1.1 — устарели, не использовать
    - Применение: HTTPS, gRPC over TLS, защищённые API, почта (SMTPS/IMAPS)
- IPsec (Internet Protocol Security)
    - Шифрование на сетевом уровне (OSI Layer 3), прозрачно для приложений
    - Режимы: Transport (только payload) / Tunnel (весь пакет + новый IP-заголовок)
    - Применение: VPN, site-to-site туннели, защита инфраструктуры

## Хеш-функции и аутентификация (Integrity & Authentication)

> Не шифруют, но обеспечивают целостность и проверку подлинности.

- SHA-256 / SHA-3
    - Криптографические хеши: детектирование изменений данных
    - Применение: цифровые подписи, HMAC, проверка целостности файлов
- HMAC (Hash-based MAC)
    - Код аутентификации сообщения на основе хеша и секретного ключа
    - Применение: JWT, API-подписи, защита вебхуков
- MD5 — криптографически сломан, не использовать для безопасности (только checksum)

## Применение

- Веб-трафик (HTTPS/API) → TLS 1.3 + AES-GCM или ChaCha20-Poly1305 + ECDHE
- Мобильные/IoT устройства → ChaCha20-Poly1305 + ECDH (меньше нагрузка на CPU)
- Внутренние m2m-сервисы → mTLS с короткоживущими сертификатами + PFS
- Сетевые туннели / VPN → IPsec (Tunnel mode) или WireGuard
- Долгосрочное хранение данных → AES-256-GCM + управление ключами (KMS/HSM)
- Подписи и аудит → ECDSA (P-256) или RSA-PSS + SHA-256

## Best Practices (чек-лист)

- Использовать TLS 1.3 (или 1.2 с PFS-шифрами), отключать SSL/early TLS
- Предпочитать AEAD-режимы (GCM, Poly1305) для шифрования + аутентификации
- Ключи RSA ≥3072 бит, ECC — P-256/P-384; регулярно ротировать
- Хранить приватные ключи в защищённом хранилище (HSM, KMS, Vault)
- Включать HSTS, OCSP Stapling, проверять цепочки сертификатов
- Не изобретать свою криптографию — использовать проверенные библиотеки (Bouncy Castle, Tink, JSSE)
- Избегать: DES, 3DES, MD5, RC4, SSL, TLS <1.2, RSA key exchange без PFS
