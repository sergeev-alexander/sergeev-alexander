# 1. Введение

> **Hamcrest** — это фреймворк для написания matcher-объектов, позволяющий декларативно определять правила «соответствия» (match rules) в тестах.
>
> Основная цель — сделать утверждения в тестах читаемыми как естественный язык, повысить гибкость проверок и улучшить сообщения об ошибках при падениях тестов.

## Зачем использовать Hamcrest?

- **Читаемость**: утверждения вида `assertThat(actual, is(equalTo(expected)))` понятнее, чем `assertEquals(expected, actual)`
- **Гибкость**: возможность комбинировать матчеры через `allOf`, `anyOf`, `not` для сложных проверок
- **Расширяемость**: простой механизм создания собственных матчеров под доменную логику
- **Универсальность**: интеграция с JUnit 3/4/5, TestNG, Mockito и другими фреймворками
- **Диагностика**: детальные сообщения об ошибках с описанием ожидаемого и фактического значения

## Основные концепции

- **`Matcher<T>`** — интерфейс, описывающий правило проверки объекта типа `T`
- **`assertThat(actual, matcher)`** — точка входа для выполнения проверки
- **Статический импорт** — ключевой приём для лаконичного синтаксиса:
  ```java
  import static org.hamcrest.MatcherAssert.assertThat;
  import static org.hamcrest.Matchers.*;
  ```
  Без него синтаксис становится многословным и теряет преимущества библиотеки
## Минимальный пример теста

```java
public class BiscuitTest {
    
    @Test
    public void testEquals() {
        Biscuit theBiscuit = new Biscuit("Ginger");
        Biscuit myBiscuit = new Biscuit("Ginger");
        assertThat(theBiscuit, equalTo(myBiscuit));
    }
}
```

В этом примере:
- `theBiscuit` — тестируемый объект (subject)
- `equalTo(myBiscuit)` — матчер, проверяющий равенство через `Object.equals()`

## Совместимость с тестовыми фреймворками

| Фреймворк | Интеграция | Примечание                                                           |
|:----------|------------|----------------------------------------------------------------------|
| JUnit 4   | Встроенная | `org.junit.Assert.assertThat` (устарел) / `MatcherAssert.assertThat` |
| JUnit 5   | Полная     | Использовать `MatcherAssert.assertThat` из Hamcrest                  |
| TestNG    | Полная     | Прямое использование `assertThat`                                    |
| Mockito   | Полная     | `Mockito.verify(obj, hamcrestMatcher)`                               |

# 2. Установка и зависимости

> Hamcrest распространяется как отдельная библиотека и не требует привязки к конкретному тестовому рантайму.
>
> Начиная с версии 2.0, все матчеры объединены в единый артефакт, что упрощает управление зависимостями и устраняет конфликты версий.

## Maven (pom.xml)

Добавьте зависимость в блок `<dependencies>` с областью `test`:

```xml
<dependency>
    <groupId>org.hamcrest</groupId>
    <artifactId>hamcrest</artifactId>
    <version>2.2</version>
    <scope>test</scope>
</dependency>
```

Для проектов, использующих JUnit 4, рекомендуется исключить устаревший `hamcrest-core`, транзитивно тянущийся из `junit:junit`, чтобы избежать `NoSuchMethodError`:

```xml
<dependency>
    <groupId>junit</groupId>
    <artifactId>junit</artifactId>
    <version>4.13.2</version>
    <scope>test</scope>
    <exclusions>
        <exclusion>
            <groupId>org.hamcrest</groupId>
            <artifactId>hamcrest-core</artifactId>
        </exclusion>
    </exclusions>
</dependency>
```

## Gradle (build.gradle / build.gradle.kts)

Аналогичная конфигурация для систем сборки на базе Gradle:

```groovy
dependencies {
    testImplementation 'org.hamcrest:hamcrest:2.2'
}
```

При использовании Kotlin DSL:

```kotlin
dependencies {
    testImplementation("org.hamcrest:hamcrest:2.2")
}
```

## Версии и артефакты

| Артефакт                        | Назначение                                      | Статус                                          |
|:--------------------------------|-------------------------------------------------|-------------------------------------------------|
| `org.hamcrest:hamcrest`         | Полная библиотека (Core, Library, Generators)   | ✅ Рекомендуется (2.x)                           |
| `org.hamcrest:hamcrest-core`    | Только базовые матчеры (`is`, `equalTo`, `not`) | ⚠️ Устарел, оставлен для обратной совместимости |
| `org.hamcrest:hamcrest-library` | Расширенные матчеры (коллекции, числа, строки)  | ⚠️ Объединён с `hamcrest` в 2.0                 |

## Конфликты зависимостей

- **JUnit 4**: по умолчанию включает `hamcrest-core 1.3`. При подключении `hamcrest 2.x` возникает конфликт классов `Matcher` и `TypeSafeMatcher`
- **Mockito**: некоторые версии Mockito могут транзитивно подтягивать старые версии Hamcrest. Используйте `dependency:tree` (Maven) или `dependencies` (Gradle) для анализа
- **Spring Boot**: BOM Spring Boot управляет версией Hamcrest. При необходимости переопределите свойство `hamcrest.version`

## Best Practices

- Явно указывайте `<scope>test</scope>` или `testImplementation`, чтобы библиотека не попадала в production-артефакт
- Не используйте `runtime` или `compile` область видимости для тестовой библиотеки
- Используйте единый артефакт `org.hamcrest:hamcrest` версии 2.x
- Не смешивайте версии 1.x и 2.x в одном проекте
- Регулярно проверяйте дерево зависимостей на наличие дубликатов `hamcrest-core`
- Не игнорируйте предупреждения `NoSuchMethodError` при запуске тестов — это почти всегда конфликт версий Hamcrest

# 3. Базовый синтаксис: assertThat

> `assertThat` — это единственный публичный метод для выполнения проверок в Hamcrest.
>
> Он принимает тестируемое значение (subject) и matcher-объект, описывающий ожидаемое состояние, формируя читаемое утверждение на естественном языке.

## Структура вызова

```java
assertThat(фактическоеЗначение, matcher);
assertThat(сообщение, фактическоеЗначение, matcher);
```

## Порядок аргументов

В отличие от классических JUnit-утверждений, в `assertThat` первым аргументом идёт **фактическое значение** (subject), а вторым — **matcher с ожидаемым состоянием**:

```java
// Hamcrest: assertThat(actual, matcher)
assertThat(user.getName(), equalTo("Alice"));

// JUnit: assertEquals(expected, actual)
assertEquals("Alice", user.getName());
```

Такой порядок логически следует структуре предложения: «Утверждаю, что **[фактическое]** соответствует **[ожидаемому]**».

## Пользовательские сообщения об ошибках

Первый (опциональный) аргумент позволяет переопределить стандартное сообщение о неудаче:

```java
assertThat("Проверка имени пользователя после регистрации", user.getName(), equalTo("Alice"));
```

При падении теста в отчёте появится именно это сообщение, что упрощает диагностику в больших сьютах.

## Сравнение с традиционными утверждениями

| Критерий            | `assertThat` (Hamcrest)                                    | `assertEquals` / `assertTrue`    |
|:--------------------|------------------------------------------------------------|----------------------------------|
| Порядок аргументов  | `actual`, `matcher`                                        | `expected`, `actual`             |
| Гибкость            | Композиция матчеров (`allOf`, `not`, `containsInAnyOrder`) | Одна проверка на вызов           |
| Читаемость          | `assertThat(list, hasSize(5))`                             | `assertEquals(5, list.size())`   |
| Сообщение об ошибке | Автоматически генерируется из структуры матчера            | Требуется ручная передача строки |

## Best Practices

✅ Делайте:

- Всегда используйте `assertThat` как единый стиль утверждений в проекте
- Не смешивайте `assertThat` и `assertEquals` в одном тестовом классе
- Не дублируйте логику проверки в сообщениях об ошибках — матчер сам сгенерирует достаточно информации
- Пишите кастомные сообщения для сложных сценариев или параметризованных тестов
- Комбинируйте статический импорт с группировкой `Matchers.*` для краткости
- Не передавайте `null` вместо subject — используйте `nullValue()` внутри matcher

# 4. Core Matchers

> **Core Matchers** — это базовый набор матчеров, входящий в ядро Hamcrest.
>
> Они покрывают 90% типичных сценариев проверки значений, ссылок, типов и состояний объектов.

## Основные матчеры

- `is()` — декоративный матчер, улучшает читаемость, не добавляет логики, просто вызывает `matches()` внутреннего матчера
- `equalTo()` — проверка равенства через `Object.equals()`, аналог `assertEquals()`
- `instanceOf()` — проверка принадлежности объекта к указанному классу или интерфейсу (аналог `instanceof`)
- `isA()` — синоним `instanceOf()`, более читаемый вариант (например, `isA(String.class)`)
- `nullValue()` / `notNullValue()` — проверка на `null` или отсутствие `null`
- `sameInstance()` — проверка ссылочного равенства (указывают ли ссылки на один и тот же объект в памяти)
- `hasToString()` — проверка результата вызова метода `toString()`
- `hasProperty()` — проверка наличия свойства (геттера) и его значения (работает через рефлексию)

## Числовые матчеры

- `closeTo(number, delta)` — проверка значения с плавающей запятой с учётом допустимой погрешности (delta)
- `greaterThan()` — проверка на больше указанного значения
- `greaterThanOrEqualTo()` — проверка на больше или равно
- `lessThan()` — проверка на меньше указанного значения
- `lessThanOrEqualTo()` — проверка на меньше или равно
- `comparesEqualTo()` — строгое равенство с использованием `compareTo()` (полезно для `BigDecimal` и `BigInteger`)

## Логические матчеры

- `allOf()` — все матчеры должны пройти (логическое И)
- `anyOf()` — хотя бы один матчер должен пройти (логическое ИЛИ)
- `not()` — инвертирует результат матчера

## Строковые матчеры

- `containsString()` — проверка, содержит ли строка подстроку
- `startsWith()` — проверка, начинается ли строка с указанного префикса
- `endsWith()` — проверка, заканчивается ли строка указанным суффиксом
- `isEmptyString()` — проверка на пустую строку (длина 0)
- `isEmptyOrNullString()` — проверка на `null` или пустую строку
- `matchesPattern()` — проверка соответствия регулярному выражению

## Коллекции и массивы

- `hasItem()` / `hasItems()` — проверка, содержит ли коллекция один или несколько элементов
- `hasSize()` — проверка размера коллекции, массива или строки
- `empty()` / `emptyArray()` / `emptyCollectionOf()` — проверка на пустоту
- `array()` — матчеры для работы с массивами
- `in()` — проверка, содержится ли элемент в указанной коллекции

## Специализированные матчеры

- `anything()` — всегда проходит проверку, полезен как заглушка или в комбинации с `anyOf()`
- `hasEntry()` — проверка наличия пары ключ-значение в `Map`
- `hasKey()` — проверка наличия ключа в `Map`
- `hasValue()` — проверка наличия значения в `Map`
- `hasItemsInArray()` — проверка наличия элементов в массиве
- `typeCompatibleWith()` — проверка, является ли тип совместимым (расширяет или реализует)

### Примеры использования

```java
// 1. is() — синтаксический сахар для читаемости
assertThat(status, is("OK"));
assertThat(status, is(equalTo("OK"))); // идентично по логике

// 2. equalTo() — проверка равенства значений
assertThat(user.getName(), equalTo("Alice"));
assertThat(list1, equalTo(list2));

// 3. instanceOf() — проверка типа
assertThat(exception, instanceOf(RuntimeException.class));
assertThat(result, instanceOf(List.class));

// 4. nullValue / notNullValue
assertThat(optimizedResult, nullValue());
assertThat(databaseConnection, notNullValue());

// 5. sameInstance() — проверка на идентичность ссылок
Object original = new Object();
Object reference = original;
assertThat(reference, sameInstance(original)); // true
```

## Сравнение Core Matchers

| Матчер           | Что проверяет                          | Аналог в JUnit               | Особенности                          |
|:-----------------|:---------------------------------------|------------------------------|--------------------------------------|
| `is()`           | Делегирует проверку вложенному матчеру | Отсутствует                  | Только для читаемости                |
| `equalTo()`      | Равенство через `.equals()`            | `assertEquals()`             | Работает с переопределённым `equals` |
| `sameInstance()` | Равенство ссылок (`==`)                | `assertSame()`               | Проверяет адрес в памяти             |
| `instanceOf()`   | Тип объекта                            | `assertTrue(x instanceof Y)` | Не бросает ошибку приведения типов   |
| `nullValue()`    | Отсутствие значения                    | `assertNull()`               | Поддерживает дженерики               |

## Best Practices

- Используйте `is()` для улучшения читаемости, но не переусердствуйте с вложенностью
- Выбирайте `equalTo()` для проверки содержимого объектов, а `sameInstance()` только когда важна идентичность ссылки
- Комбинируйте `instanceOf()` с последующей проверкой кастованных полей через `allOf()`
- Не проверяйте `null` через `equalTo(null)` — используйте явный `nullValue()` для точных сообщений об ошибках

# 5. Matchers для коллекций

> **Matchers для коллекций** — мощный набор инструментов для проверки List, Set, Map, Iterable и массивов.
>
> Позволяют проверять содержимое, порядок, размер и наличие элементов без ручных циклов и вспомогательных методов.

## Основные матчеры

- `hasSize()` — проверка размера коллекции или массива
- `hasItem()` / `hasItems()` — проверка наличия одного или нескольких элементов (порядок не важен)
- `contains()` — строгая проверка порядка и полного совпадения элементов
- `containsInAnyOrder()` — проверка полного совпадения элементов без учёта порядка
- `empty()` / `emptyIterable()` / `emptyArray()` — проверка на пустоту
- `hasEntry()` / `hasKey()` / `hasValue()` — специализированные матчеры для Map
- `arrayContaining()` / `arrayContainingInAnyOrder()` — аналоги для примитивных и объектных массивов

## Примеры использования

```java
List<String> fruits = List.of("apple", "banana", "cherry");
Map<String, Integer> scores = Map.of("Alice", 95, "Bob", 88);

// Размер
assertThat(fruits, hasSize(3));

// Наличие элементов
assertThat(fruits, hasItem("banana"));
assertThat(fruits, hasItems("apple", "cherry"));

// Строгий порядок
assertThat(fruits, contains("apple", "banana", "cherry"));

// Любой порядок, но полное совпадение
assertThat(fruits, containsInAnyOrder("cherry", "apple", "banana"));

// Проверка Map
assertThat(scores, hasKey("Alice"));
assertThat(scores, hasValue(88));
assertThat(scores, hasEntry("Bob", 88));

// Массивы
String[] arr = {"x", "y"};
assertThat(arr, arrayContaining("x", "y"));
assertThat(arr, arrayContainingInAnyOrder("y", "x"));
```

## Best Practices

- Используйте `contains()` когда порядок элементов критичен для бизнес-логики
- Не используйте `contains()` для unordered collections (Set, HashMap keySet) — тест будет падать нестабильно
- Выбирайте `containsInAnyOrder()` для проверки результатов запросов без гарантии сортировки
- Не путайте `hasItems()` (проверка подмножества) с `containsInAnyOrder()` (полное совпадение)
- Не передавайте `null` в матчеры коллекций без предварительной проверки `nullValue()` или `notNullValue()`
- Комбинируйте `hasItems()` с другими матчерами: `assertThat(list, hasItems(is("A"), not("B")))`
  ```java
  // Проверяем, что список содержит "Apple" и НЕ содержит "Grape"
        assertThat(fruits, hasItems(
            is("Apple"),           // точное совпадение
            not("Grape")           // отрицание
        ));
  ```

# 6. Matchers для строк

> **Matchers для строк** предоставляют специализированные инструменты для проверки текстовых данных.
>
> Они позволяют проверять содержимое, регистр, пробелы, вхождения подстрок и соответствие регулярным выражениям без ручного написания вспомогательной логики.

## Основные матчеры

- `startsWith()` — проверка, начинается ли строка с указанного префикса
- `endsWith()` — проверка, заканчивается ли строка указанным суффиксом
- `containsString()` — проверка наличия подстроки в любом месте
- `equalToIgnoringCase()` — проверка полного равенства без учёта регистра
- `equalToIgnoringWhiteSpace()` — проверка равенства с игнорированием лишних пробелов и переносов строк
- `matchesPattern()` / `matchesRegex()` — проверка соответствия регулярному выражению

## Примеры использования

```java
String message = "  Hello, World!  ";

// Проверка вхождений
assertThat(message, containsString("World"));
assertThat(message, startsWith("  Hello"));
assertThat(message, endsWith("World!  "));

// Игнорирование регистра
assertThat("HELLO", equalToIgnoringCase("hello"));

// Игнорирование пробелов и переносов
assertThat("  Hello   World  ", equalToIgnoringWhiteSpace("Hello World"));

// Регулярные выражения
assertThat("user123@example.com", matchesPattern("^[\\w.-]+@[\\w.-]+\\.\\w+$"));
assertThat("123-45-6789", matchesRegex("\\d{3}-\\d{2}-\\d{4}"));
```

## Best Practices

✅ Делайте:

- Используйте `containsString()` вместо `assertTrue(text.contains(...))` для более понятных отчётов об ошибках
- Применяйте `equalToIgnoringWhiteSpace()` при сравнении HTML, XML или многострочных логов
- Не используйте `equalToIgnoringWhiteSpace()` для строгих проверок форматов, где пробелы значимы
- Комбинируйте `matchesPattern()` с валидацией форматов (email, телефоны, UUID)
- Не пишите сложные регулярные выражения без тестовых примеров — используйте `assertThat` с несколькими кейсами
- Не игнорируйте кодировку и локали при сравнении строк, если они зависят от региональных настроек

# 7. Matchers для чисел

> **Matchers для чисел** предназначены для безопасного и точного сравнения числовых значений.
>
> Они решают проблему точности сравнения чисел с плавающей запятой, проверяют диапазоны и порядок следования значений.

## Основные матчеры

- `closeTo(number, delta)` — проверка значения с плавающей запятой с учётом допустимой погрешности (delta)
- `greaterThan()` / `lessThan()` — проверка на больше/меньше указанного значения
- `greaterThanOrEqualTo()` / `lessThanOrEqualTo()` — проверка на больше/меньше или равно
- `comparesEqualTo()` — строгое равенство с использованием `compareTo()` а не `equals()` (полезно для `BigDecimal`, `BigInteger`, дат, кастомных числовых типов)

## Примеры использования

```java
double pi = 3.14159;
int count = 42;
BigDecimal price = new BigDecimal("19.99");

// Проверка с допуском (для float/double)
assertThat(pi, closeTo(3.14, 0.002));

// Сравнение величин
assertThat(count, greaterThan(30));
assertThat(count, lessThanOrEqualTo(42));

// Строгое сравнение через compareTo (для BigDecimal)
assertThat(price, comparesEqualTo(new BigDecimal("19.990")));

// Проверка диапазона (комбинация)
assertThat(count, allOf(greaterThanOrEqualTo(40), lessThanOrEqualTo(50)));
```

## Best Practices

✅ Делайте:

- Всегда используйте `closeTo()` для всех сравнений `double` и `float`, чтобы избежать ложных падений из-за точности IEEE 754
- Не используйте `equalTo()` для чисел с плавающей запятой без явного контроля точности
- Применяйте `comparesEqualTo()` для `BigDecimal`, так как `equalTo()` может не учитывать математическую эквивалентность (например, `1.0` vs `1.00`)
- Комбинируйте матчеры через `allOf()` для проверки попадания значения в допустимый диапазон
- Не проверяйте диапазоны через `assertTrue(val >= min && val <= max)` — теряется читаемость и детализация ошибок
  Диапазоны через композицию — `allOf(greaterThanOrEqualTo(min), lessThanOrEqualTo(max))` даёт читаемый и информативный тестовый код

# 8. Логические комбинации

> **Логические комбинации** позволяют объединять несколько матчеров в одно сложное утверждение.
>
> Они реализуют булеву логику (AND, OR, NOT) прямо внутри `assertThat`, избавляя от вложенных условий и множественных проверок в одном тесте.

## Основные операторы

- `allOf()` — логическое И: все переданные матчеры должны выполниться успешно
- `anyOf()` — логическое ИЛИ: достаточно выполнения хотя бы одного матчера
- `not()` — логическое НЕ: инвертирует результат проверки вложенного матчера
- `either()` / `both()` — плавный (fluent) интерфейс для комбинации ровно двух условий

## Примеры использования

```java
String payload = "{\"status\": \"OK\", \"code\": 200, \"data\": []}";

// allOf: все условия должны быть истинными
assertThat(payload, allOf(
    containsString("\"status\": \"OK\""),
    containsString("\"code\": 200"),
    not(containsString("error"))
));

// anyOf: достаточно одного совпадения
assertThat(httpStatus, anyOf(is(200), is(201), is(204)));

// not: отрицание проверки
assertThat(errorLog, not(containsString("NullPointerException")));

// Fluent-интерфейс для двух условий
assertThat(age, either(greaterThan(18)).or(equalTo(18))); // age >= 18
assertThat(user, both(isActive()).and(hasRole("admin")));
```

## Сравнение логических операторов

| Оператор        | Логика | Количество аргументов | Поведение при сбое                                     |
|:----------------|--------|:---------------------:|--------------------------------------------------------|
| `allOf()`       | AND    |          2+           | Останавливается на первом несовпадении (short-circuit) |
| `anyOf()`       | OR     |          2+           | Останавливается на первом совпадении                   |
| `not()`         | NOT    |           1           | Инвертирует результат вложенного матчера               |
| `either().or()` | OR     |           2           | Fluent-вариант `anyOf()` для двух условий              |
| `both().and()`  | AND    |           2           | Fluent-вариант `allOf()` для двух условий              |

## Best Practices

- Используйте `allOf()` для валидации объектов, где важно проверить несколько независимых полей одновременно
- Применяйте `anyOf()` при тестировании кодов ответов API или альтернативных состояний системы
- Не используйте `anyOf()` для проверок, где важны все условия одновременно — это маскирует частичные сбои
- Комбинируйте `not()` с `containsString()` или `empty()` для проверки отсутствия нежелательных значений или ошибок в логах
- Не забывайте, что `allOf()` и `anyOf()` используют short-circuit вычисление: порядок матчеров напрямую влияет на сообщение об ошибке
  ```java
  // Порядок от дешёвых к дорогим проверкам
  assertThat(value, anyOf(
        equalTo(0),                      // ✅ Быстрая проверка
        equalTo(42),                     // ✅ Дёшево
        equalTo(getExpensiveResource())  // Выполнится ТОЛЬКО если все предыдущие не прошли
  ));
  
  // Проверяем все условия
  // ❌ Плохо: при падении не поймёте, какие ещё условия нарушены
  assertThat(value, allOf(
        greaterThan(30),     // ❌ FAIL
        lessThan(20),        // ⚠️ НЕ выполнится
        equalTo(25)          // ⚠️ НЕ выполнится
  ));
  ```

# 9. Создание кастомных Matchers

> **Кастомные матчеры** позволяют инкапсулировать сложную доменную логику проверок в переиспользуемые компоненты.
>
> Hamcrest предоставляет базовые классы, которые берут на себя boilerplate-код, 
> оставляя разработчику только реализацию логики соответствия и форматирования сообщений об ошибках.

## Когда создавать кастомный матчер

- **Повторяющаяся логика** — одна и та же проверка встречается в нескольких тестах
- **Сложные доменные правила** — проверка требует комбинации полей объекта или внешних вычислений
- **Улучшение диагностики** — стандартные матчеры дают общие сообщения, а кастомный покажет конкретное поле и ожидаемое значение
- **Сокрытие деталей** — тесты читаются как бизнес-требования, без технических подробностей реализации проверки

## Базовые классы для расширения

- `TypeSafeMatcher<T>` — рекомендуемая основа, автоматически обрабатывает `null` и проверяет тип объекта
- `BaseMatcher<T>` — низкоуровневый интерфейс, требует ручной проверки типов и `null`
- `FeatureMatcher<T, U>` — удобен для проверки конкретного поля/метода объекта через делегирование другому матчеру
- `Description` — интерфейс для построения читаемых сообщений об ожидаемом результате и фактическом несоответствии

## Шаблон реализации TypeSafeMatcher

```java
public class IsAdminUser extends TypeSafeMatcher<User> {

    @Override
    protected boolean matchesSafely(User user) {
        return user.isActive() && "ADMIN".equals(user.getRole());
    }

    @Override
    public void describeTo(Description description) {
        description.appendText("активного пользователя с ролью ADMIN");
    }

    @Override
    protected void describeMismatchSafely(User user, Description mismatchDescription) {
        mismatchDescription.appendText("пользователь имел статус ")
                           .appendValue(user.isActive())
                           .appendText(" и роль ")
                           .appendValue(user.getRole());
    }

    // Фабричный метод для статического импорта
    public static IsAdminUser isAdminUser() {
        return new IsAdminUser();
    }
}
```
## Описание ошибок: describeTo и describeMismatch

- `describeTo()` — формирует сообщение о том, **что ожидалось**. Вызывается при падении теста
- `describeMismatchSafely()` — формирует сообщение о том, **что фактически было получено**. 
  
  Позволяет указать конкретные поля, не совпавшие с ожиданием
- Методы `appendText()` и `appendValue()` автоматически форматируют строки и экранируют специальные символы
- Переопределение обоих методов критично для понятных отчётов в CI/CD пайплайнах

## Использование в тестах

```java
@Test
public void testCustomMatcher() {
    User user = new User("admin01", false, "MODERATOR");
    assertThat(user, isAdminUser());
    // При падении:
    // Expected: активного пользователя с ролью ADMIN
    //      but: пользователь имел статус <false> и роль <"MODERATOR">
}
```
## Сравнение подходов к созданию матчеров

| Подход                              | Сложность   | Безопасность типов  | Когда использовать                           |
|:------------------------------------|-------------|---------------------|----------------------------------------------|
| `TypeSafeMatcher`                   | Низкая      | ✅ Автоматическая    | Большинство кастомных проверок               |
| `BaseMatcher`                       | Высокая     | ❌ Ручная проверка   | Совместимость со старым кодом                |
| `FeatureMatcher`                    | Средняя     | ✅ Частичная         | Проверка одного поля через встроенный матчер |
| Lambda-матчеры (JUnit 5 + Hamcrest) | Минимальная | ✅ Зависит от лямбды | Быстрые одноразовые проверки                 |

## Best Practices

- Не размещайте логику создания матчера внутри тестовых методов — это нарушает принцип DRY
- Выносите кастомные матчеры в отдельный пакет `matchers` или `hamcrest`
- Всегда добавляйте статический фабричный метод для использования без `new`
- Переопределяйте `describeMismatchSafely` для точной диагностики в отчётах
- Используйте `TypeSafeMatcher` как стандартный выбор для новых матчеров
- Не дублируйте встроенные матчеры (`containsString`, `greaterThan`) без веской доменной причины
- Не используйте `BaseMatcher` без необходимости — ручная проверка `null` и типов увеличивает риск `ClassCastException`
