# Functional Testing Types (Функциональные типы тестирования)

> Функциональное тестирование - это проверка соответствия функциональности системы заявленным требованиям.

<details>
  <summary>
    <b>Security Testing (Тестирование безопасности)</b>
  </summary>

> `Security Testing` (Тестирование безопасности): Проверка способности системы противостоять злонамеренным попыткам получить доступ или вызвать сбой.

## Основные понятия

- `Security Testing` (Тестирование безопасности) - Вид тестирования, направленный на выявление уязвимостей, угроз и рисков в программном обеспечении с целью предотвращения несанкционированного доступа, потери данных, дохода или репутации.
- `CIA Triad` (Триада CIA) - Три основополагающих принципа информационной безопасности:
    - `Confidentiality` (Конфиденциальность) - Защита информации от несанкционированного раскрытия.
    - `Integrity` (Целостность) - Гарантия того, что данные могут быть изменены только уполномоченными субъектами.
    - `Availability` (Доступность) - Обеспечение доступа к ресурсам авторизованным пользователям по запросу.

## Цели и задачи

- `Vulnerability Identification` (Выявление уязвимостей) - Обнаружение слабых мест в системе, которые могут быть использованы злоумышленниками.
- `Threat Assessment` (Оценка угроз) - Анализ потенциальных источников атак и их возможного воздействия.
- `Risk Mitigation` (Смягчение рисков) - Разработка и рекомендация мер для снижения вероятности и последствий инцидентов безопасности.
- `Compliance Verification` (Проверка соответствия) - Убедиться, что система соответствует отраслевым стандартам и нормативным требованиям.

## Основные виды уязвимостей

- `Cross-Site Scripting` (`XSS`) (Межсайтовый скриптинг) - Уязвимость веб-приложений, позволяющая внедрять и выполнять вредоносные клиентские скрипты в браузере жертвы через уязвимые веб-страницы.
- `Cross-Site Request Forgery` (`CSRF` / `XSRF`) (Подделка межсайтовых запросов) - Уязвимость, при которой злоумышленник заставляет аутентифицированного пользователя выполнить нежелательное действие на доверенном сайте без его ведома.
- `Code Injection` (Инъекция кода) - Класс уязвимостей, позволяющий внедрять и исполнять произвольный код (например, SQL, OS-команды, PHP, JavaScript) на целевой системе. Включает:
    - `SQL Injection` (SQL-инъекция) - Внедрение вредоносных SQL-запросов для несанкционированного доступа или манипуляции данными.
    - `OS Command Injection` (Инъекция команд ОС) - Выполнение произвольных команд операционной системы от имени уязвимого приложения.
- `Server-Side Includes Injection` (`SSI Injection`) (Инъекция серверных включений) - Уязвимость, позволяющая внедрять и выполнять серверные команды через директивы SSI в HTML-документах.
- `Authorization Bypass` (Обход авторизации) - Уязвимость, позволяющая получить доступ к ресурсам или функциям, предназначенным для других пользователей или ролей, минуя механизмы контроля доступа.
- `Insecure Direct Object References` (`IDOR`) (Небезопасные прямые ссылки на объекты) - Уязвимость, возникающая при использовании предсказуемых идентификаторов (например, в URL или параметрах), что позволяет получить доступ к данным других пользователей.
- `Security Misconfiguration` (Неправильная конфигурация безопасности) - Уязвимость, вызванная ошибками в настройке серверов, фреймворков, баз данных или приложений (например, открытые директории, дефолтные пароли, избыточные функции).
- `Broken Authentication` (Нарушенная аутентификация) - Уязвимости в механизмах входа в систему, такие как слабые пароли, отсутствие блокировки после неудачных попыток, небезопасное управление сессиями.
- `Sensitive Data Exposure` (Раскрытие конфиденциальных данных) - Уязвимость, связанная с недостаточной защитой чувствительных данных (паролей, платежных реквизитов) при хранении или передаче (например, отсутствие шифрования).
- `Path Traversal` / `Directory Traversal` (Обход пути / Директорий) - Уязвимость, позволяющая читать файлы за пределами целевой директории на сервере с помощью специальных последовательностей (например, `../`).

## Основные типы тестирования безопасности

- `Vulnerability Scanning` (Сканирование на уязвимости) - Автоматизированный поиск известных сигнатур уязвимостей в системе.
- `Security Scanning` (Сканирование безопасности) - Выявление слабых мест в сети и системах с последующими рекомендациями по их устранению; может быть ручным или автоматическим.
- `Penetration Testing` (`Pen Testing`) (Тестирование на проникновение) - Имитация атаки реального злоумышленника для оценки защищённости системы.
- `Risk Assessment` (Оценка рисков) - Классификация выявленных рисков (низкий, средний, высокий) и предложение процедур по их снижению.
- `Security Auditing` (Аудит безопасности) - Внутренняя проверка приложений и операционных систем, включая анализ исходного кода на наличие проблем безопасности.
- `Ethical Hacking` (Этичный хакинг) - Законная деятельность специалистов («white hat»), направленная на выявление уязвимостей до их обнаружения злоумышленниками.
- `Posture Assessment` (Оценка состояния безопасности) - Комплексный анализ, объединяющий сканирование, этичный хакинг и оценку рисков для определения общего уровня безопасности организации.

## Методологии и подходы

- `Black Box Testing` (Метод «чёрного ящика») - Тестирование без знания внутренней структуры системы; имитирует внешнюю атаку.
- `White Box Testing` (Метод «белого ящика») - Тестирование с полным доступом к исходному коду и архитектуре системы.
- `Gray Box Testing` (Модель «серого ящика») - Комбинированный подход с частичным знанием внутренней структуры системы.
- `Tiger Box` (Tiger Box) - Использование ноутбука с набором различных ОС и инструментов для взлома для проведения оценки уязвимостей.

## Ключевые аспекты тестирования безопасности

- `Authentication Testing` (Тестирование аутентификации) - Проверка механизмов входа в систему, включая:
    - Правила сложности пароля
    - Блокировку после неудачных попыток (защита от `Brute-Force Attack` / Атаки грубой силы)
    - Функциональность восстановления пароля и выхода из системы
- `Authorization Testing` (Тестирование авторизации) - Проверка прав доступа ролей и пользователей к функциям и данным, включая:
    - `Horizontal Privilege Escalation` (Горизонтальное повышение привилегий)
    - `Path Traversal` (Обход пути)
    - Доступ к модулям, недоступным для данной роли
- `Data Protection` (Защита данных) - Обеспечение безопасности данных в трёх состояниях:
    - При хранении (шифрование конфиденциальных данных в БД)
    - При передаче (использование HTTPS вместо HTTP)
    - При обработке (корректная расшифровка в месте назначения)
- `Input Validation` (Валидация ввода) - Защита от атак типа:
    - `SQL Injection` (SQL-инъекция)
    - `Cross-Site Scripting` (`XSS`) (Межсайтовый скриптинг)
    - Ограничение длины и типа вводимых данных, запрет HTML/скриптовых тегов
- `Session Management` (Управление сессией) - Проверка:
    - Истечения срока действия сессии при простое
    - Корректного завершения сессии при выходе
    - Безопасности и срока жизни cookies
    - Возможности одновременных сессий одного пользователя
- `Error Handling` (Обработка ошибок) - Убедиться, что сообщения об ошибках и трассировки стека не раскрывают конфиденциальную информацию.
- `Secure File Upload` (Безопасная загрузка файлов) - Проверка ограничений по типу, размеру файла и сканирование на вирусы.
- `Payment Security` (Безопасность платежей) - Особое внимание к защите от инъекций, безопасному хранению криптографических ключей и переполнению буфера.

## Роли в тестировании безопасности

- `Ethical Hacker` / `White Hat` (Этичный хакер) - Специалист, проводящий тестирование с разрешения владельца системы.
- `Black Hat Hacker` (Хакер-преступник) - Лицо, совершающее несанкционированный взлом в корыстных целях.
- `Script Kiddie` («Скрипт-кидд») - Неопытный злоумышленник, использующий готовые инструменты без глубокого понимания.

## Интеграция в SDLC

- `Requirements Phase` - Анализ требований на безопасность и определение сценариев злоупотребления.
- `Design Phase` - Проведение анализа рисков и разработка плана тестирования безопасности.
- `Coding & Unit Testing` - Применение статического и динамического анализа безопасности (SAST/DAST).
- `Integration & System Testing` - Тестирование методом «чёрного ящика» и сканирование на уязвимости.
- `Implementation` - Проведение тестирования на проникновение и финального сканирования.
- `Support` - Анализ воздействия патчей и обновлений на безопасность.

## Преимущества

- `Early Risk Detection` (Раннее выявление рисков) - Снижение стоимости исправления дефектов при внедрении на ранних этапах SDLC.
- `Proactive Defense` (Превентивная защита) - Позволяет устранить уязвимости до их эксплуатации.
- `Developer Guidance` (Направление для разработчиков) - Помогает команде писать более безопасный код.

## Основные понятия

- `UI Testing` (Тестирование пользовательского интерфейса) - Проверка корректности отображения, навигации, взаимодействия и соответствия макетам графических элементов интерфейса.
- `UX Testing` (Тестирование пользовательского опыта) - Оценка удобства, эффективности, удовлетворенности и интуитивности взаимодействия пользователя с системой для достижения своих целей.

## Цели тестирования UI/UX

- `Usability Validation` (Валидация удобства использования) - Подтверждение, что интерфейс позволяет пользователю легко и эффективно выполнять задачи.
- `Consistency Verification` (Проверка согласованности) - Обеспечение единообразия визуального стиля, терминологии и поведения элементов по всему приложению.
- `Accessibility Compliance` (Соответствие требованиям доступности) - Проверка, что интерфейс может использоваться людьми с ограниченными возможностями (в соответствии со стандартами WCAG).
- `Cross-Platform & Cross-Browser Compatibility` (Кроссплатформенная и кроссбраузерная совместимость) - Гарантия корректного отображения и функционирования на различных устройствах, ОС и браузерах.
- `User Journey Validation` (Валидация пользовательских сценариев) - Проверка полноты и эффективности ключевых путей пользователя от начала до завершения цели.

## Ключевые принципы проектирования UI/UX

- `KISS Principle` (`Keep It Simple, Stupid`) (Принцип простоты) - Интерфейс должен быть максимально простым и избегать ненужной сложности.
- `Don't Make Me Think` (Не заставляй меня думать!) - Дизайн должен быть интуитивно понятным, чтобы пользователь не тратил усилия на интерпретацию.
- `Remove the Obvious` (Удаляем очевидное) - Исключение избыточных подсказок и элементов, которые не добавляют ценности.
- `Signal-to-Noise Ratio` (Соотношение сигнала и шума) - Максимальное усиление полезной информации и минимизация визуального «мусора».
- `Functionality Over Fashion` (Лучше рабочее, чем модное) - Приоритет функциональности и удобства перед эстетикой или трендами.
- `Familiar Controls` (Знакомые элементы управления) - Использование стандартных, узнаваемых компонентов (кнопки, чекбоксы, иконки), снижающих порог входа.
- `Users Don't Read, They Scan` (Люди не читают, они сканируют) - Предпочтение визуальной иерархии, иконок, заголовков и интерактивных элементов длинному тексту.
- `Principle of Borrowing` (Принцип заимствования) - Использование проверенных паттернов и решений вместо изобретения новых без необходимости.
- `Miller's Law` (Закон Миллера) - Ограничение количества одновременно представленных элементов выбора или информации до 5±2 единиц для улучшения восприятия.
- `Gestalt Principles of Grouping` (Принципы группировки по Гештальту) - Организация информации в логические блоки с помощью близости, схожести, замыкания и других визуальных приёмов.
- `Intuitive Clarity` (Интуитивная ясность) - Элементы интерфейса (кнопки, ссылки, разделы) должны быть легко обнаруживаемыми и их назначение — очевидным.
- `Visibility of Key Functions` (Все полезное на виду) - Критически важные действия и информация должны быть видимы без необходимости поиска или дополнительных кликов.
- `Three-Click Rule` (Правило трёх кликов) - Пользователь должен иметь возможность достичь любой цели в системе не более чем за три действия.
- `Consistency` (Однородность) - Единый стиль, цветовая палитра, типографика, поведение элементов и терминология по всему продукту.
- `Problem-Solving Focus` (Фокус на решении проблем) - Продукт должен помогать пользователю достигать его целей, а не создавать новые трудности.
- `Prevention of Accidental Actions` (Защита от случайных действий) - Реализация подтверждений, отмены действий и защитных механизмов против ошибок (например, удаление, отправка).
- `Unity of Control` (Принцип единства управления) - Предоставление централизованных настроек и контроля над функциями, если это улучшает управляемость.
- `Trend Awareness` (Учет тенденций) - Следование современным практикам дизайна для обеспечения актуальности и ожидаемого пользователем опыта, но без слепого копирования.

## Стандарты и руководства

- `ISO 9241-110` (Эргономические принципы взаимодействия человек-система) - Международный стандарт, описывающий диалоговые принципы: пригодность для задачи, самоописываемость, соответствие ожиданиям, обучаемость, восстанавливаемость ошибок, индивидуализируемость и соответствие пользовательским характеристикам.
- `WCAG` (`Web Content Accessibility Guidelines`) (Руководство по доступности веб-контента) - Набор рекомендаций для обеспечения доступности сайтов и приложений для людей с ограниченными возможностями.
- `Platform-Specific Guidelines` (Платформенные гайдлайны) - Официальные рекомендации от разработчиков платформ:
    - `Material Design` (Google)
    - `Human Interface Guidelines` (Apple)
    - `Fluent Design System` (Microsoft)

## Методы тестирования UI/UX

- `Heuristic Evaluation` (Эвристическая оценка) - Экспертный анализ интерфейса на соответствие установленным принципам (эвристикам).
- `Usability Testing` (Юзабилити-тестирование) - Наблюдение за реальными пользователями при выполнении задач в интерфейсе.
- `A/B Testing` (A/B-тестирование) - Сравнение двух или более вариантов интерфейса для определения наиболее эффективного.
- `Eye Tracking` (Айтрекинг) - Анализ фокуса внимания пользователя с помощью специального оборудования.
- `Accessibility Audits` (Аудит доступности) - Проверка интерфейса на соответствие стандартам WCAG с использованием автоматизированных и ручных методов.

    <b>Internationalization (I18N)</b>
  </summary>

> `Internationalization` (`I18N`) (Интернационализация) - Процесс проектирования и разработки программного обеспечения таким образом, чтобы оно было независимо от языка, региона и культурных особенностей, что позволяет легко адаптировать его для любой локали без изменения исходного кода.

## Цели тестирования I18N

- `Cultural Neutrality` (Культурная нейтральность) - Обеспечение того, что приложение не содержит хардкода, зависящего от конкретной культуры или языка.
- `Resource Externalization` (Вынесение ресурсов) - Проверка, что все локализуемые элементы (текст, изображения, форматы) вынесены во внешние файлы (локали).
- `Locale Independence` (Независимость от локали) - Подтверждение, что основной код приложения функционирует одинаково при любом выбранном языке или регионе.

## Ключевые области проверки

- `Hard-coded Strings` (Хардкод строк) - Поиск текстовых констант, вшитых в код, которые не могут быть переведены.
- `Date/Time/Number/Currency Formatting` (Форматирование данных) - Проверка корректной обработки различных форматов даты, времени, чисел и валют через системные API.
- `Character Encoding` (Кодировка символов) - Обеспечение поддержки Unicode (UTF-8) для корректного отображения всех языков, включая CJK (китайский, японский, корейский).
- `Text Expansion & Contraction` (Расширение и сжатие текста) - Убедиться, что интерфейс устойчив к изменению длины текста (например, немецкий текст может быть на 30% длиннее английского).
- `Bi-directional (BiDi) Support` (Поддержка двунаправленного текста) - Готовность интерфейса к работе с языками, пишущимися справа налево (RTL), такими как арабский или иврит.
- `Sorting & Collation` (Сортировка) - Проверка алгоритмов сортировки, учитывающих правила конкретного языка.
- `Input Method Editors (IME)` (Редакторы методов ввода) - Поддержка сложных систем ввода для языков с большим количеством иероглифов.
- `Installation & Configuration` (Установка и настройка) - Возможность установки и настройки приложения в любой локальной среде.

    <b>Localization (L10N)</b>
  </summary>

> `Localization` (`L10N`) (Локализация) - Процесс адаптации глобализированного программного обеспечения для конкретного региона или языка путем добавления локальных компонентов и перевода контента с учетом культурных особенностей.

## Цели тестирования L10N

- `Linguistic Accuracy` (Лингвистическая точность) - Корректность перевода, грамматики, терминологии и стиля.
- `Cultural Appropriateness` (Культурная уместность) - Отсутствие изображений, символов, цветов или контента, оскорбительных или непонятных в данной культуре.
- `Functional Equivalence` (Функциональный эквивалент) - Сохранение полной функциональности продукта после локализации.

## Ключевые области проверки

- `UI Layout & Truncation` (Макет интерфейса) - Отсутствие наложения текста, обрезки строк и смещения элементов из-за изменения длины переведенного текста.
- `Regional Formats` (Региональные форматы) - Корректное отображение:
    - Даты (ДД/ММ/ГГГГ vs ММ/ДД/ГГГГ)
    - Времени (12-часовой vs 24-часовой формат)
    - Чисел (разделители тысяч и десятичных знаков)
    - Валют (символ, позиция, формат)
    - Адресов и телефонных номеров
- `Graphics & Icons` (Графика и иконки) - Отсутствие текста в изображениях, культурно нейтральные символы, корректные изображения для региона.
- `Hotkeys & Shortcuts` (Горячие клавиши) - Адаптация сочетаний клавиш под раскладку и терминологию целевого языка.
- `Documentation & Help` (Документация) - Перевод и актуальность всех справочных материалов.
- `SEO & Meta Tags` (Мета-теги) - Корректное заполнение title, description и keywords на целевом языке.
- `Pseudo-localization Readiness` (Готовность к псевдо-локализации) - Способность системы обрабатывать строки с удлинением, специальными символами и RTL-метками для предварительного тестирования.

</details>

  </summary>

## Основные понятия

- `Globalization` (`G11N`) (Глобализация) - Стратегический процесс, объединяющий интернационализацию (I18N) и локализацию (L10N), направленный на подготовку и выпуск продукта для мирового рынка.

## Цели тестирования G11N

- `Market Readiness` (Готовность к выходу на рынок) - Обеспечение того, что продукт соответствует всем техническим, юридическим и культурным требованиям целевых регионов.
- `Scalability for New Locales` (Масштабируемость для новых локалей) - Легкость добавления поддержки новых языков и регионов без значительных переделок.
- `Consistent Global Experience` (Единый глобальный опыт) - Предоставление пользователям по всему миру качественного и согласованного опыта взаимодействия с продуктом.

## Ключевые области проверки

- `End-to-End Locale Switching` (Полное переключение локали) - Возможность пользователя сменить язык/регион в настройках и получить полностью адаптированный интерфейс и контент.
- `Compliance with Local Regulations` (Соответствие местному законодательству) - Проверка на соответствие требованиям GDPR (ЕС), CCPA (Калифорния), законам о защите данных и рекламе в конкретных странах.
- `Local Payment & Shipping Methods` (Местные способы оплаты и доставки) - Интеграция и тестирование популярных в регионе платежных систем и служб доставки.
- `Time Zone Handling` (Обработка часовых поясов) - Корректное отображение и расчет времени в зависимости от местоположения пользователя.
- `Performance Across Regions` (Производительность по регионам) - Обеспечение приемлемой скорости работы приложения для пользователей в разных географических точках (CDN, локальные серверы).
- `Accessibility Standards` (Стандарты доступности) - Соответствие местным стандартам доступности (например, WCAG, Section 508).

</details>

- `API Testing` (Тестирование `API`): Тестирование программных интерфейсов приложения (слой сервисов) без графического интерфейса (`GUI`).
- `Database` / `DB Testing` (Тестирование баз данных): Проверка целостности данных (`data integrity`), корректности запросов (`queries`), производительности БД.

  <summary>
    <b>Integration Testing (Тестирование взаимодействия / Интеграционное)</b>
  </summary>

## Основные понятия

- `Interoperability Testing` — функциональное тестирование, направленное на проверку способности компонентов или систем корректно взаимодействовать друг с другом в рамках заданной среды.
- Включает в себя элементы **интеграционного тестирования** и **тестирования совместимости**, но фокусируется именно на корректности обмена данными, вызовов и согласованности поведения между разными системами.

## Цели и задачи

- Убедиться, что система корректно обменивается данными с внешними компонентами (API, базы данных, сторонние сервисы и т.д.).
- Проверить соблюдение стандартов взаимодействия (протоколы, форматы данных, кодировки, версии интерфейсов).
- Обеспечить стабильность работы при изменении одной из взаимодействующих систем.
- Подтвердить, что интеграция не требует значительных доработок со стороны любой из сторон.

## Области применения

- Интеграция с внешними API (платежные системы, SSO, CRM и др.).
- Обмен данными между микросервисами.
- Совместная работа клиентского и серверного приложений разных версий.
- Взаимодействие с устаревшими (legacy) системами.
- Кросс-платформенные решения (например, мобильное приложение + веб-бэкенд).

## Ключевые аспекты проверки

- Корректность форматов и структур передаваемых данных (JSON, XML, protobuf и др.).
- Согласованность версий интерфейсов и протоколов.
- Обработка ошибок при недоступности или некорректном ответе внешней системы.
- Безопасность взаимодействия (аутентификация, авторизация, шифрование).
- Производительность и отказоустойчивость при сетевых задержках или частичной недоступности.

## Метрики оценки

- Количество необходимых модификаций для успешной интеграции.
- Время настройки и подключения новой системы.
- Частота сбоев при взаимодействии в реальных условиях.
- Соответствие отраслевым или внутренним стандартам совместимости.

## Лучшие практики

- Использовать контрактное тестирование (например, Pact) для фиксации соглашений между системами.
- Автоматизировать сценарии взаимодействия как часть CI/CD.
- Поддерживать актуальную документацию по API и протоколам обмена.
- Проводить тестирование с реальными версиями внешних систем или их точными моками.
- Учитывать обратную совместимость при выпуске новых версий интерфейсов.

</details>

  <summary>
    <b>Contract Testing (Контрактное тестирование)</b>
  </summary>

## Основные понятия

- `Contract Testing` (Контрактное тестирование) - Метод тестирования взаимодействия между независимыми компонентами (например, микросервисами), основанный на заранее согласованном контракте, описывающем формат и поведение запросов и ответов.
- `Integration Point` (Точка интеграции) - Интерфейс, через который два или более сервиса обмениваются данными.
- `Consumer` (Потребитель) - Сервис или клиент, который инициирует запрос к другому сервису для получения данных или выполнения действия.
- `Provider` (Поставщик) - Сервис, который получает запрос от потребителя и возвращает соответствующий ответ.

## Цели и задачи

- `Compatibility Assurance` (Гарантия совместимости) - Обеспечение того, что изменения в одном сервисе не нарушат работу других, зависящих от него сервисов.
- `Early Integration Issue Detection` (Раннее выявление проблем интеграции) - Обнаружение несоответствий в интерфейсах на ранних этапах разработки, до запуска дорогостоящих сквозных тестов.
- `Decoupled Development` (Независимая разработка) - Позволяет командам разрабатывать и выпускать свои сервисы независимо, опираясь на стабильный контракт.
- `Documentation Synchronization` (Синхронизация документации) - Контракт служит «живой» документацией, которая всегда актуальна, так как используется в тестах.

## Типы контрактного тестирования

- `Consumer-Driven Contract Testing` (`CDC`) (Контрактное тестирование, управляемое потребителем) - Потребитель определяет, какие именно запросы он будет отправлять и какие ответы ожидает. Поставщик обязуется удовлетворять этим ожиданиям.
- `Provider-Driven Contract Testing` (Контрактное тестирование, управляемое поставщиком) - Поставщик публикует свой контракт (например, в виде спецификации OpenAPI), и все потребители должны ему соответствовать.

## Ключевые артефакты

- `Contract File` (Файл контракта) - Сериализованный файл (обычно в формате JSON), содержащий описание всех допустимых запросов и ожидаемых ответов для конкретной точки интеграции.

## Преимущества

- `Fast Feedback Loop` (Быстрый цикл обратной связи) - Тесты выполняются быстро, так как не требуют развертывания всей системы.
- `High Reliability & Low Flakiness` (Высокая надежность и низкая «хрупкость») - Изолированные тесты менее подвержены случайным сбоям по сравнению со сквозными.
- `Cost Efficiency` (Экономическая эффективность) - Снижает затраты на инфраструктуру и обслуживание тестов.
- `Safe Independent Releases` (Безопасные независимые релизы) - Команды могут выпускать свои сервисы с уверенностью, что не сломают интеграцию.
- `Clear Service Boundaries` (Четкие границы сервисов) - Помогает явно определить, какие данные и функции являются частью публичного API.

## Ограничения

- `Partial Behavior Coverage` (Частичное покрытие поведения) - Контрактное тестирование проверяет только интерфейс и формат данных, но не бизнес-логику или побочные эффекты (например, изменение состояния в БД).
- `Mock Fidelity` (Верность моков) - Поведение мока (заглушки) может не полностью соответствовать реальному сервису в production-среде.
- `Contract Maintenance Overhead` (Накладные расходы на поддержку контрактов) - Требует дисциплины от команд для своевременного обновления контрактов при изменении API.

## Когда использовать

- При работе с архитектурой на основе микросервисов.
- Для тестирования взаимодействия с внешними API.
- Когда сквозные интеграционные тесты медленные, хрупкие и дорогие в поддержке.
- Для обеспечения стабильности в распределенных командах, работающих над разными частями системы.

## Способы проведения

- `Testing Against a Deployed Service` (Тестирование на развернутом сервисе) - Запуск контрактных тестов против реально работающего экземпляра поставщика.
- `Testing Against a Mock/Stub` (Тестирование на моке/заглушке) - Использование генерируемых на основе контракта моков для изолированного тестирования потребителя.
- `Test-Driven Contract Development` (Разработка контракта через тестирование) - Написание контрактных тестов до реализации самого сервиса (в духе TDD).

</details>

<details>
  <summary>
    <b>Regression Testing (Регрессионное тестирование)</b>
  </summary>

## Основные понятия

- `Regression Testing` (Регрессионное тестирование) - Тип функционального тестирования, направленный на подтверждение того, что недавно внесённые изменения в код (исправления, новые функции, оптимизация) не вызвали непреднамеренных побочных эффектов и не нарушили существующую функциональность.

## Цели и задачи

- `Side Effect Detection` (Обнаружение побочных эффектов) - Выявление дефектов, которые появились в ранее работающем коде из-за изменений в другом месте системы.
- `Stability Assurance` (Гарантия стабильности) - Обеспечение того, что основной функционал системы остаётся работоспособным после каждого изменения.
- `Confidence in Releases` (Уверенность в релизах) - Предоставление команде разработки и заинтересованным сторонам уверенности в том, что новый выпуск не сломает существующие возможности продукта.

## Триггеры для проведения

- Исправление дефектов (bug fixes).
- Добавление новой функциональности.
- Изменение требований к существующей функции.
- Рефакторинг или оптимизация исходного кода.
- Применение патчей и обновлений зависимостей.
- Изменения в конфигурации системы или окружения.

## Стратегии отбора тестов

- `Full Regression` (Полный регресс) - Запуск всего набора тестов. Используется при крупных изменениях или перед важными релизами.
- `Partial / Selective Regression` (Частичный / Избирательный регресс) - Запуск только тех тестов, которые связаны с изменёнными модулями и их зависимостями.
- `Prioritized Regression` (Приоритезированный регресс) - Запуск тестов в порядке их важности (например, сначала критические и высоко приоритетные).

## Подходы к организации

- `Test Suite Maintenance` (Поддержка тестового набора) - Регулярное обновление и актуализация тест-кейсов в соответствии с изменениями в требованиях и функционале.
- `Impact Analysis` (Анализ влияния) - Определение областей системы, которые могут быть затронуты внесёнными изменениями, для фокусировки тестирования.
- `Smoke & Sanity Testing` (Дымовое и sanity-тестирование) - Быстрая проверка основных функций перед запуском полного регрессионного цикла.

## Преимущества автоматизации

- `Speed` (Скорость) - Автоматизированные тесты выполняются значительно быстрее ручных.
- `Repeatability` (Повторяемость) - Гарантирует одинаковые условия выполнения при каждом запуске.
- `Cost Efficiency` (Экономическая эффективность) - Снижает долгосрочные затраты на повторяющиеся проверки, особенно в проектах с частыми релизами.
- `CI/CD Integration` (Интеграция с CI/CD) - Позволяет запускать регрессионные тесты автоматически при каждом коммите или сборке, обеспечивая быструю обратную связь.

## Лучшие практики

- `Automate Repetitive Tests` (Автоматизируйте повторяющиеся тесты) - Все тесты, которые нужно запускать часто, должны быть автоматизированы.
- `Maintain a Living Test Repository` (Поддерживайте актуальный репозиторий тестов) - Удаляйте устаревшие тесты и добавляйте новые по мере развития продукта.
- `Combine Manual and Automated` (Комбинируйте ручное и автоматизированное тестирование) - Используйте ручное тестирование для исследовательских и UX-аспектов, а автоматизацию — для проверки стабильности.
- `Run Tests Early and Often` (Запускайте тесты как можно раньше и чаще) - Интегрируйте регрессионное тестирование в ежедневный процесс разработки.

</details>

<details>
  <summary>
    <b>End-to-End Testing (Сквозное тестирование)</b>
  </summary>

> Системное тестирование фокусируется на тестировании всей программной системы на соответствие спецификациям требований, включая функциональные и нефункциональные аспекты. Выполняется до сквозного тестирования.
>
> Сквозное тестирование охватывает систему и все её взаимодействия с внешними подсистемами и интерфейсами.

## Основные понятия

- `End-to-End Testing` / `E2E Testing` (Сквозное тестирование) - Вид тестирования, направленный на проверку программного обеспечения от начала до конца, включая его интеграцию с внешними интерфейсами, подсистемами и базами данных для валидации полного производственного сценария.
- `Chain Testing` (Цепочечное тестирование) - Синоним сквозного тестирования, подчеркивающий последовательную проверку взаимосвязанных компонентов системы.
- `Production Scenario` (Производственный сценарий) - Реалистичный поток операций, имитирующий использование системы в реальных условиях эксплуатации.

## Цели и задачи

- `System Flow Validation` (Валидация системного потока) - Проверка корректности и целостности всего процесса обработки данных через все компоненты системы.
- `Dependency Verification` (Проверка зависимостей) - Обеспечение работоспособности системы при взаимодействии со всеми внутренними и внешними зависимостями.
- `Data Integrity Assurance` (Обеспечение целостности данных) - Гарантия того, что данные не искажаются и корректно передаются между всеми этапами и системами.
- `Risk Mitigation` (Смягчение рисков) - Снижение риска отказа всей системы из-за сбоя в одной из её взаимосвязанных подсистем.

## Процесс и методология

- `Test Environment Setup` (Настройка тестовой среды) - Подготовка окружения, максимально приближенного к production, для проведения тестов.
- `Requirements Traceability` (Сквозное отслеживание требований) - Установление и поддержание связи между исходными требованиями, тест-кейсами и результатами тестирования.
- `User Function Definition` (Определение пользовательских функций) - Идентификация и описание ключевых функций системы, их входных/выходных данных и взаимосвязей.
- `Condition Building` (Построение условий) - Формулирование набора условий (последовательность, временные рамки, состояния данных) для каждой пользовательской функции.
- `Test Case Development` (Разработка тест-кейсов) - Создание одного или нескольких тест-кейсов для каждого определенного тестового сценария на основе условий.

## Метрики и отслеживание прогресса

- `Test Case Preparation Status` (Статус подготовки тест-кейсов) - Метрика, отражающая реальный прогресс подготовки тест-кейсов по сравнению с запланированным.
- `Weekly Testing Progress` (Еженедельный прогресс тестирования) - Процент завершенных тестов за неделю относительно общего плана, включая статистику по успешным, неудачным и незапущенным тестам.
- `Defect Status and Details` (Статус и детали дефектов) - Отчет о количестве открытых и закрытых дефектов, а также их распределении по серьезности и приоритету.
- `Environment Availability` (Доступность среды) - Метрика, показывающая отношение фактического времени доступности тестовой среды к запланированному.

## Инструменты автоматизации

- `testRigor` (Инструмент testRigor) - Платформа для кодлесс-автоматизации сквозного тестирования, использующая генеративный ИИ для создания и поддержки тестов на естественном языке.
- `Codeless Test Automation` (Кодлесс-автоматизация) - Подход, позволяющий создавать и поддерживать тесты без написания кода, часто с использованием естественного языка.
- `Cross-Platform Coverage` (Кроссплатформенное покрытие) - Возможность одного теста охватывать веб, мобильные приложения (нативные и гибридные), API и другие каналы взаимодействия (email, SMS).
