# NoSQL Базы Данных

## Что такое NoSQL

- **Определение**: NoSQL (Not Only SQL) — тип баз данных, использующий гибкую модель хранения и обработки данных,
  отличную от традиционных реляционных (SQL) систем.
- **Назначение**: Работа с большими объемами данных (Big Data), высокая скорость обработки, решение задач
  масштабируемости.
- **Ключевые особенности**:
    - **Гибкость структуры**: Не требуется фиксированная схема данных (schema-less). Записи могут иметь разное
      количество полей.
    - **Форматы хранения**: Данные хранятся в зависимости от типа БД (документы, столбцы, пары ключ-значение, графы).
    - **Масштабируемость**: Ориентированы на горизонтальное масштабирование (распределение по нескольким
      серверам/кластерам).
    - **Согласованность**: Часто используют модель BASE вместо строгих транзакций ACID.

## Типы NoSQL Баз Данных

### 1. Документно-ориентированные (Document-Oriented)

- **Примеры**: MongoDB, CouchDB, Firebase.
- **Описание**: Данные хранятся в виде документов (обычно JSON, BSON, XML). Каждый документ — это самодостаточная
  единица данных.
- **Основные свойства и механика**:
    - **Гибкая схема**: Поля могут отличаться в разных документах одной коллекции. Структура может меняться со временем.
    - **Репликация (Replica Set)**:
        - В любой момент времени есть один главный узел (Primary), который принимает все запросы на запись.
        - Остальные узлы (Secondary) — это его "тени". Они только читают.
        - Если главный узел падает, остальные узлы выбирают себе нового главного.
        - **Цель**: Защищает от потери данных и позволяет читать с копий (распределение нагрузки на чтение).
    - **Шардинг (Sharding)**:
        - Данные распределяются по множеству машин (шардам) по заданному полю (шард-ключу).
        - **Цель**: Горизонтальное масштабирование хранения и производительности записи.

### 2. Колоночно-ориентированные (Column-Family)

- **Примеры**: Cassandra, HBase, ScyllaDB.
- **Описание**: Данные хранятся в виде столбцов, а не строк. Эффективно для работы с огромными объемами данных и
  аналитики.
- **Основные свойства и механика**:
    - **Семейства столбцов**: Данные группируются логически вместе, что ускоряет чтение конкретных атрибутов.
    - **Оптимизация записи**: Высокая скорость записи благодаря структуре хранения (часто используют LSM-деревья).
    - **Настраиваемая согласованность**: Можно выбирать уровень согласованности для каждого запроса (например, кворум
      записи/чтения).
    - **Отказоустойчивость**: Данные автоматически реплицируются на несколько узлов, нет единой точки отказа.

### 3. Ключ-Значение (Key-Value)

- **Примеры**: Redis, DynamoDB, Riak, Memcached.
- **Описание**: Данные хранятся как пары "ключ-значение". Ключ уникален, значение может быть любым (строка, объект,
  бинарные данные).
- **Основные свойства и механика**:
    - **Скорость доступа**: Поиск по ключу происходит крайне быстро (часто O(1)), многие БД работают в оперативной
      памяти (In-Memory).
    - **Простота**: Отсутствие сложных запросов и JOIN операций. Основные операции: get, set, delete.
    - **TTL (Time To Live)**: Возможность автоматического удаления данных через заданное время.
    - **Использование**: Кэширование, хранение сессий, корзины покупок, очередей задач.

### 4. Графовые (Graph)

- **Примеры**: Neo4j, ArangoDB, Amazon Neptune.
- **Описание**: Данные хранятся в виде графов, состоящих из узлов (сущности), ребер (связи) и свойств.
- **Основные свойства и механика**:
    - **Связи как данные**: Связи между объектами хранятся явно и являются объектами первого класса.
    - **Обход графа (Traversal)**: Очень быстрый поиск путей и связей между удаленными узлами (например, "друзья
      друзей").
    - **Отсутствие JOIN**: Не требуются дорогостоящие операции соединения таблиц, так как связи физически сохранены.
    - **Использование**: Социальные сети, рекомендательные системы, обнаружение мошенничества, сетевые топологии.

## Когда использовать NoSQL

- **Динамическая структура**: Данные часто меняются, и их схема не фиксирована.
- **Большие объемы**: Требуется масштабирование на множество серверов (горизонтальное).
- **Высокая скорость**: Важна низкая задержка доступа к данным (кэш, реальное время).
- **Распределенная архитектура**: Требуется высокая отказоустойчивость и доступность (AP по теореме CAP).

## Модели согласованности

- **ACID (Традиционные SQL)**:
    - Атомарность, Согласованность, Изоляция, Долговечность.
    - Гарантия целостности данных в любой момент времени.
- **BASE (NoSQL)**:
    - **Basically Available**: Система доступна даже при частичных сбоях.
    - **Soft State**: Состояние системы может меняться со временем даже без ввода новых данных (из-за асинхронной
      репликации).
    - **Eventual Consistency**: Данные станут согласованными через некоторое время, но не гарантированно мгновенно.
