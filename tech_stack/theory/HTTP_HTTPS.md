HTTP - Протокол прикладного уровня для передачи данных в клиент-серверной архитектуре

Клиент отправляет запрос - сервер возвращает ответ. 

Сервер не хранит состояние (stateless).

HTTPS - это тот же HTTP, но «упакованный» в TLS-шифрование, чтобы никто по пути не подсмотрел и не подменил данные.

HTTPS шифрует трафик, проверяет подлинность сервера через сертификат и следит за целостностью данных.

Когда клиент подключается по HTTPS:

- сначала происходит handshake (рукопожатие): стороны договариваются о параметрах шифрования, сервер предъявляет
  сертификат, клиент его проверяет.
- Дальше весь обмен идёт в зашифрованном виде.
- В современных системах используется Forward Secrecy: для каждой сессии ключ создаётся заново, так что даже при утечке
  долгосрочного ключа прошлые сессии остаются в безопасности.

## HTTP (HyperText Transfer Protocol)

> Протокол прикладного уровня для передачи данных в клиент-серверной архитектуре
>
> Stateless: каждый запрос независим, сервер не хранит состояние между запросами

- ### Версии:
    - HTTP/1.1: текстовый, pipelining (ограниченно), keep-alive, доминирует в legacy
    - HTTP/2: бинарный, мультиплексирование, HPACK-сжатие заголовков, server push (опционально)
    - HTTP/3: поверх QUIC (UDP), решает проблему head-of-line blocking, быстрее при потере пакетов
- ### Структура запроса/ответа:
    - Start line: метод + URI + версия (запрос) / статус-код + фраза (ответ)
    - Headers: метаданные (Content-Type, Authorization, Cache-Control и др.)
    - Body: опциональные данные (JSON, form-data, binary)
- ### Основные HTTP-методы:
    - GET: получение ресурса (идемпотентный, безопасный)
    - POST: создание/отправка данных (не идемпотентный)
    - PUT: полная замена ресурса (идемпотентный)
    - PATCH: частичное обновление (не всегда идемпотентный)
    - DELETE: удаление ресурса (идемпотентный)
    - HEAD/OPTIONS: метаданные и поддержка CORS preflight
- ### Коды статусов (группы):
    - 2xx Success: 200 OK, 201 Created, 204 No Content
    - 3xx Redirection: 301 Moved Permanently, 304 Not Modified (кэш)
    - 4xx Client Error: 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 429 Too Many Requests
    - 5xx Server Error: 500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable
- Важные концепции:
    - Идемпотентность: повтор одного запроса не меняет результат (GET/PUT/DELETE - да, POST - нет)
    - Кэширование: заголовки Cache-Control, ETag, Last-Modified
    - Cookies: механизм хранения состояния на клиенте (Session, JWT), флаги Secure/HttpOnly/SameSite

### HTTPS (HTTP Secure)

> HTTP поверх TLS/SSL: шифрование + аутентификация + целостность данных
>
> Порт по умолчанию: 443 (vs 80 для HTTP)

- ### Что обеспечивает TLS:
    - Конфиденциальность: трафик нечитаем для посторонних (симметричное шифрование)
    - Целостность: данные не могут быть незаметно изменены (HMAC/AEAD)
    - Аутентификация: клиент уверен, что общается с нужным сервером (сертификаты X.509)
- ### Упрощённая схема TLS Handshake:
    1. ClientHello: клиент предлагает версии TLS, шифры, random
    2. ServerHello: сервер выбирает параметры, отправляет свой сертификат
    3. Верификация: клиент проверяет сертификат (цепочка до доверенного CA, срок, имя домена)
    4. Key Exchange: через (EC)DHE стороны вырабатывают общий сессионный ключ (Forward Secrecy)
    5. Finished: обмен зашифрованными confirmation-сообщениями - начало защищённой сессии
- ### Сертификаты и PKI:
    - Формат X.509, выпускаются Certificate Authorities (Let's Encrypt, DigiCert, и др.)
    - Types: DV (domain validation), OV (organization), EV (extended validation)
    - Проверка: цепочка доверия, CRL/OCSP для проверки отзыва, pinning (опционально)
- ### Ключевые улучшения в TLS 1.3:
    - Убраны уязвимые алгоритмы (RSA key exchange, CBC, SHA-1, RC4)
    - Обязательный Forward Secrecy (только ECDHE)
    - Handshake за 1 RTT (и 0-RTT для повторных соединений, с оговорками по безопасности)
    - Шифрование большей части handshake-сообщений

## HTTP vs HTTPS: ключевые отличия

| Аспект                 | HTTP                        | HTTPS                                                   |
|------------------------|-----------------------------|---------------------------------------------------------|
| Шифрование             | Нет                         | Да (TLS)                                                |
| Порт                   | 80                          | 443                                                     |
| Аутентификация сервера | Нет                         | Да (сертификат)                                         |
| Целостность данных     | Нет гарантии                | Гарантирована (MAC/AEAD)                                |
| SEO и браузеры         | Помечается как "Not Secure" | Требуется для современных фич (HTTP/2, Service Workers) |
| Производительность     | HTTP/1.1 медленнее          | HTTP/2/3 + TLS 1.3 = часто быстрее HTTP/1.1             |

### Best Practices (чек-лист для production)

- Всегда использовать HTTPS в production, редирект HTTP → HTTPS
- Принудительно TLS 1.2/1.3, отключить старые версии и слабые шифры
- Включить HSTS (Strict-Transport-Security) для защиты от downgrade-атак
- Настроить безопасные cookies: Secure, HttpOnly, SameSite=Strict/Lax
- Использовать современные заголовки безопасности:
    - Content-Security-Policy, X-Content-Type-Options, X-Frame-Options, Referrer-Policy
- Регулярно обновлять сертификаты, использовать автоматизацию (Let's Encrypt + certbot)
- Для внутренних сервисов: mTLS (взаимная аутентификация) + короткоживущие сертификаты
- Мониторить истечение сертификатов и уязвимости (SSL Labs test, CVE)

### Частые ошибки и анти-паттерны

- Mixed Content: HTTPS-страница загружает ресурсы по HTTP - браузер блокирует или предупреждает
- Игнорирование проверки сертификата в коде (trust all) - уязвимость MITM
- Хранение чувствительных данных в URL (логи, рефереры) - использовать headers/body
- Отсутствие валидации входных данных даже поверх HTTPS - шифрование ≠ защита от инъекций
- Использование self-signed сертификатов в публичных сервисах - только для dev/test

---

### Важные security-заголовки:

- Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
- Content-Security-Policy: default-src 'self'
- X-Content-Type-Options: nosniff
- X-Frame-Options: DENY
- Referrer-Policy: strict-origin-when-cross-origin
