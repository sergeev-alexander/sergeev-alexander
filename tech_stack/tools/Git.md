## Что такое Git

**Git** - распределённая система контроля версий (VCS) для отслеживания изменений в коде. 
Позволяет нескольким разработчикам работать над одним проектом, сохраняя полную историю изменений.

### Ключевые особенности:

- Распределённая архитектура - у каждого разработчика полная копия репозитория
- Быстрое ветвление и слияние
- Работа офлайн (локальные операции)
- Стандарт индустрии для управления кодом

---

## Основные концепции

- `Репозиторий` - Место хранения проекта со всей историей изменений
- `Коммит` - Зафиксированное состояние проекта в момент времени
- `Ветка` - Независимая линия разработки
- `Staging Area` - Промежуточная зона для подготовки изменений к коммиту
- `Remote` - Удалённый репозиторий (GitHub, GitLab, Bitbucket)
- `Merge` - Объединение изменений из нескольких веток
- `Pull Request` / `Merge Request` - Запрос на слияние изменений для проверки

## Инициализация и клонирование

### Создание репозитория:

- `git init` - Инициализирует новый локальный репозиторий в текущей директории
- `git clone <url>` - Клонирует удалённый репозиторий на локальную машину
- `git clone <url> <folder>` - Клонирует в указанную папку

### Примеры:

- `git init` - Создать новый репозиторий
- `git clone https://github.com/user/project.git` - Клонировать проект
- `git clone git@github.com:user/project.git` - Клонировать через SSH

## Просмотр статуса и изменений

### Статус и различия:

- `git status` - Показывает статус файлов (изменённые, добавленные, не отслеживаются)
- `git diff` - Показывает различия между рабочим каталогом и последним коммитом
- `git diff --staged` - Показывает различия в staging area
- `git diff <branch1> <branch2>` - Сравнение двух веток
- `git log` - Показывает историю коммитов
- `git log --oneline` - Краткая история коммитов
- `git log --graph` - История с графом веток
- `git show <commit>` - Показать детали коммита

### Примеры:

- `git status` - Проверить текущий статус
- `git diff` - Посмотреть изменения перед коммитом
- `git log --oneline -10` - Последние 10 коммитов

## Добавление и фиксация изменений

### Staging и Commit:

- `git add <file>` - Добавить конкретный файл в staging area
- `git add .` - Добавить все изменённые файлы
- `git add -A` - Добавить все изменения (включая удалённые)
- `git commit -m "сообщение"` - Зафиксировать изменения с сообщением
- `git commit -am "сообщение"` - Добавить и закоммитить изменённые файлы
- `git commit --amend` - Изменить последний коммит (сообщение или файлы)
- `git commit --amend --no-edit` - Изменить коммит без изменения сообщения
- `git reset HEAD <file>` - Убрать файл из staging area
- `git restore --staged <file>` - Убрать файл из staging
- `git restore <file>` - Отменить изменения в файле

### Примеры:

- `git add src/main.java` - Добавить конкретный файл
- `git add .` - Добавить все файлы
- `git commit -m "Add user authentication"` - Закоммитить с сообщением
- `git commit --amend -m "Fix typo in message"` - Исправить сообщение коммита

### Хорошие сообщения коммитов:

- `feat: add login functionality` - Новая функциональность
- `fix: resolve null pointer exception` - Исправление бага
- `docs: update README` - Обновление документации
- `refactor: simplify user service` - Рефакторинг
- `test: add unit tests for auth` - Добавление тестов

## Работа с ветками

### Управление ветками:

- `git branch` - Показать все локальные ветки
- `git branch -a` - Показать все ветки (локальные + удалённые)
- `git branch <name>` - Создать новую ветку
- `git branch -d <name>` - Удалить ветку (безопасно)
- `git branch -D <name>` - Удалить ветку (принудительно)
- `git checkout <branch>` - Переключиться на ветку
- `git checkout -b <branch>` - Создать и переключиться на ветку
- `git switch <branch>` - Переключиться на ветку
- `git switch -c <branch>` - Создать и переключиться
- `git merge <branch>` - Слить ветку с текущей
- `git rebase <branch>` - Перебазировать текущую ветку на другую
- `git mergetool` - Запустить инструмент для разрешения конфликтов

### Примеры:

- `git branch` - Список веток
- `git branch feature/login` - Создать ветку feature/login
- `git checkout feature/login` - Переключиться на ветку
- `git checkout -b feature/register` - Создать и переключиться
- `git merge feature/login` - Слить feature/login в текущую
- `git branch -d feature/login` - Удалить ветку после слияния

### Merge vs Rebase:

- `git merge` - Создаёт merge commit, сохраняет историю обеих веток - Для слияния feature в main
- `git rebase` - Переписывает историю, создаёт линейную цепочку - Для обновления feature ветки из main

Пример:

Исходное состояние:

```text
main:     A---B---C
               \
feature:        D---E
```

`git merge` (сохраняет историю веток):

```text
main:     A---B---C---F
               \     /
feature:        D---E
```

`git rebase` (создаёт линейную историю):

```text
main:        A---B---C

feature:     A---B---C---D'---E'
            (переписанные коммиты)
            
После rebase можно сделать fast-forward merge и история станет линейной.
Информация о том, что работа велась в отдельной ветке, потеряется:

main:     A---B---C---D'---E'
```

## Удалённые репозитории

### Remote операции:

- `git remote -v` - Показать список удалённых репозиториев
- `git remote add origin <url>` - Добавить удалённый репозиторий
- `git remote remove origin` - Удалить удалённый репозиторий
- `git remote set-url origin <url>` - Изменить URL remote
- `git fetch <remote>` - Получить изменения из remote (без слияния)
- `git fetch --all` - Получить изменения со всех remote
- `git pull <remote> <branch>` - Получить и слить изменения
- `git pull --rebase` - Получить и перебазировать изменения
- `git push <remote> <branch>` - Отправить изменения в remote
- `git push -u origin <branch>` - Отправить и установить upstream
- `git push --force` - Принудительный push (осторожно!)
- `git push --force-with-lease` - Безопасный принудительный push

### Примеры:

- `git remote -v` - Проверить remote
- `git remote add origin https://github.com/user/project.git` - Добавить origin
- `git fetch origin` - Получить изменения
- `git pull origin main` - Получить и слить main
- `git push origin feature/login` - Отправить ветку
- `git push -u origin feature/login` - Отправить и установить tracking

## История и откат изменений

### Просмотр истории:

- `git log` - История коммитов
- `git log --oneline` - Краткая история
- `git log --graph --oneline --all` - Граф всех веток
- `git log --author="name"` - Коммиты автора
- `git log --since="2026-01-01"` - Коммиты с даты
- `git log <file>` - История изменений файла

### Откат изменений:

- `git reset <commit>` - Мягкий сброс (изменения в working directory)
- `git reset --soft <commit>` - Сброс с сохранением в staging
- `git reset --hard <commit>` - Жёсткий сброс (все изменения удаляются)
- `git revert <commit>` - Создать коммит, отменяющий изменения
- `git revert HEAD~3..HEAD` - Отменить последние 3 коммита
- `git cherry-pick <commit>` - Применить конкретный коммит к текущей ветке

### Примеры:

- `git reset HEAD~1` - Отменить последний коммит (сохранить изменения)
- `git reset --hard HEAD~1` - Отменить последний коммит (удалить изменения)
- `git revert abc1234` - Отменить конкретный коммит новым коммитом
- `git cherry-pick abc1234` - Взять коммит из другой ветки

## Теги

### Управление тегами:

- `git tag` - Показать все теги
- `git tag <name>` - Создать тег на текущем коммите
- `git tag -a <name> -m "сообщение"` - Создать аннотированный тег
- `git show <tag>` - Показать информацию о теге
- `git push origin <tag>` - Отправить тег в remote
- `git push origin --tags` - Отправить все теги
- `git tag -d <name>` - Удалить локальный тег
- `git push origin --delete <tag>` - Удалить remote тег

### Примеры:

- `git tag v1.0.0` - Создать тег версии
- `git tag -a v1.0.0 -m "Release version 1.0"` - Аннотированный тег
- `git push origin v1.0.0` - Отправить тег
- `git push origin --tags` - Отправить все теги

## Дополнительные команды

### Stash (временное сохранение):

- `git stash` - Сохранить изменения без коммита
- `git stash list` - Показать список stash
- `git stash apply` - Применить последний stash
- `git stash apply stash@{n}` - Применить конкретный stash
- `git stash pop` - Применить и удалить stash
- `git stash drop` - Удалить stash
- `git stash clear` - Очистить все stash

### Конфликты:

- `git status` - Показать файлы с конфликтами
- `git diff` - Показать конфликты в файлах
- `git add <file>` - Пометить конфликт как разрешённый
- `git merge --abort` - Прервать слияние
- `git rebase --abort` - Прервать rebase
- `git rebase --continue` - Продолжить rebase после разрешения конфликтов

### Очистка и обслуживание:

- `git clean -n` - Показать неудаляемые файлы (dry run)
- `git clean -f` - Удалить неудаляемые файлы
- `git gc` - Запустить garbage collection
- `git fsck` - Проверить целостность репозитория

### Полезные алиасы (добавить в .gitconfig):

```ignorelang
    [alias]
        co = checkout
        br = branch
        ci = commit
        st = status
        lg = log --oneline --graph --all
        unstage = reset HEAD --
        last = log -1 HEAD
```

## Стратегии для автотестов

### Модели репозиториев:

| Модель                | Описание                                 | Когда использовать                   |
|:----------------------|------------------------------------------|--------------------------------------|
| Единый репозиторий    | Автотесты в том же репозитории что и код | Маленькие команды, тесная интеграция |
| Отдельный репозиторий | Автотесты в отдельном репозитории        | Разные команды, разные циклы релизов |

### Ветвление для автотестов:

- `main/master` - Стабильная версия тестов
- `develop` - Ветка разработки
- `feature/<name>` - Ветка для новой функциональности
- `hotfix/<name>` - Ветка для срочных исправлений
- `release/<version>` - Ветка подготовки релиза

### Рекомендации:

- Синхронизируйте ветки тестов с ветками приложения
- Используйте Pull Requests для всех изменений в тестах
- Интегрируйте тесты в CI/CD pipeline
- Тегируйте версии тестов под релизы приложения
- Храните тестовые данные в репозитории (без чувствительных данных)
- Используйте код-ревью для тестового кода

### Пример workflow:

1. Создать ветку `feature/new-test` от `develop`
2. Разработать тесты в ветке
3. Создать Pull Request
4. Пройти код-ревью
5. Запустить CI/CD pipeline
6. Слить в `develop` после approval
7. При релизе - слить в `main` и создать тег

## Шпаргалка по основным командам

### Начало работы:

- `git init` - Новый репозиторий
- `git clone <url>` - Клонировать репозиторий
- `git remote add origin <url>` - Добавить remote

### Ежедневная работа:

- `git status` - Проверить статус
- `git add .` - Добавить изменения
- `git commit -m "msg"` - Закоммитить
- `git pull origin main` - Получить изменения
- `git push origin main` - Отправить изменения

### Ветки:

- `git branch` - Список веток
- `git checkout -b <name>` - Создать и переключиться
- `git merge <branch>` - Слить ветку
- `git branch -d <name>` - Удалить ветку

### Откат:

- `git reset --soft HEAD~1` - Отменить коммит (сохранить)
- `git reset --hard HEAD~1` - Отменить коммит (удалить)
- `git revert <commit>` - Отменить коммит новым коммитом

### Аварийные:

- `git stash` - Временно сохранить изменения
- `git merge --abort` - Прервать слияние
- `git reflog` - История всех действий (спасение)
