## Что такое CI/CD

CI/CD — набор практик и инструментов для автоматизации разработки, тестирования и доставки приложений.

### Простыми словами:

Раньше:
- `Разработчик пишет код` → `ждёт неделю до релиза` → `вручную тестируют` → `выкладывают на сервер` → `находят баги` → 
  `исправляют` → `снова релиз`

С CI/CD:
- `Разработчик пишет код` → `автоматически тестируется` → `автоматически проверяется` → `автоматически выкладывается` → 
  `баги находятся сразу`

### Зачем это нужно:

| Проблема без CI/CD             | Решение с CI/CD             |
|:-------------------------------|-----------------------------|
| Ошибки находят через недели    | Ошибки находят через минуты |
| Релизы раз в месяц             | Релизы каждый день          |
| Ручное тестирование            | Автоматическое тестирование |
| Страх деплоить                 | Деплой без страха           |
| Конфликты между разработчиками | Конфликты решаются сразу    |

## CI vs CD vs Continuous Delivery

Три понятия которые часто путают:

### Continuous Integration (CI) — Непрерывная интеграция

- **Что:** Автоматическая сборка и тестирование кода при каждом коммите
- **Когда:** При каждом push в репозиторий
- **Зачем:** Убедиться что код работает и тесты проходят

**Пример:**
`Разработчик сделал push` → `GitLab запустил пайплайн` → `Сборка` → `Unit тесты` → `Интеграционные тесты` → `Отчёт`

**Результат:** Вы знаете что код рабочий через 5-10 минут

---

### Continuous Delivery — Непрерывная доставка

- **Что:** Автоматическая подготовка кода к релизу
- **Когда:** После успешного CI
- **Зачем:** Код всегда готов к деплою, но деплой ручной

**Пример:**
`CI прошёл` → `Сборка Docker образа` → `Загрузка в Nexus` → `Ожидание кнопки "Deploy"`

**Результат:** Можно выкатить релиз в любой момент одной кнопкой

---

### Continuous Deployment — Непрерывное развёртывание

- **Что:** Автоматический деплой на продакшн
- **Когда:** После успешного Delivery
- **Зачем:** Полная автоматизация без ручного вмешательства

**Пример:**
`Все тесты прошли` → `Автоматический деплой на prod` → `Мониторинг здоровья приложения`

**Результат:** Пользователи получают обновления автоматически

---

### Визуальное сравнение:

```text
    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
    │    Commit   │───▶│      CI     │───▶│  Delivery   │───▶│  Deployment │
    │             │    │  (сборка    │    │  (готово    │    │  (автомат   │
    │             │    │   + тесты)  │    │   к релизу) │    │   на prod)  │
    └─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
                              │                  │                  │
                         Автоматически     Автоматически     Автоматически
                             ✅                 ✅                 ✅
                         
    Continuous Delivery:     ✅                 ✅                 🖱️ (ручная кнопка)
    Continuous Deployment:   ✅                 ✅                 ✅
```

## Стенды и окружения

Стенд — изолированная среда для работы с приложением.

### Типы стендов:

| Стенд    | Назначение   | Ветки         | Данные     | Кто использует |
|:---------|--------------|---------------|------------|----------------|
| dev      | Разработка   | feature/*     | Тестовые   | Разработчики   |
| test     | Тестирование | develop       | Тестовые   | Тестировщики   |
| pre-prod | Предрелиз    | main/pre-prod | Копия prod | Заказчики, QA  |
| prod     | Продакшн     | main/release  | Реальные   | Пользователи   |

### Подробно по каждому:

#### Dev (Development)

**Цель:** Быстрая проверка кода разработчиком

**Что развёрнуто:**

- Feature ветки (незавершённый функционал)
- Часто ломается
- Нет стабильных данных

**Домен:** `dev.my-app.com`

**Когда использовать:**

- Проверить что код компилируется
- Быстро протестировать фичу
- Отладка в среде близкой к серверу

---

#### Test (Testing)

**Цель:** Полноценное тестирование функционала

**Что развёрнуто:**

- Develop ветка (проверенные фичи)
- Стабильная версия для тестов
- Тестовые данные

**Домен:** `test.my-app.com`

**Когда использовать:**

- Регрессионное тестирование
- Запуск автотестов
- Проверка новых фич тестировщиками

---

#### Pre-Prod (Demo/Staging)

**Цель:** Финальная проверка перед релизом

**Что развёрнуто:**

- Main или pre-prod ветка
- Все проверенные фичи
- Данные копируются с prod

**Домен:** `pre-prod.my-app.com` или `demo.my-app.com`

**Когда использовать:**

- Демонстрация заказчику
- Финальное тестирование
- Проверка производительности

---

#### Prod (Production)

**Цель:** Работа с реальными пользователями

**Что развёрнуто:**

- Main или release ветка
- Только стабильный код
- Реальные данные пользователей

**Домен:** `my-app.com`

**Когда использовать:**

- Только релизные версии
- После approval от всех команд

---

### Почему нельзя всё делать локально?

| Проблема локальной разработки     | Решение через стенды                      |
|:----------------------------------|-------------------------------------------|
| Другая версия БД                  | Стенд имеет ту же БД что и prod           |
| Нет доступа к внешним сервисам    | Стенд подключён ко всем сервисам          |
| Невозможно тестировать с командой | Общий стенд для всей команды              |
| Не видно проблем сети             | Стенд имеет реальную сетевую конфигурацию |
| Нет нагрузки                      | Стенд можно нагрузить для тестов          |

### Пример доступа к стендам:

```text
    Проект: my-super-app

    Prod:     https://my-super-app/api/v1/search
    Pre-Prod: https://pre-prod.my-super-app/api/v1/search
    Test:     https://test.my-super-app/api/v1/search
    Dev:      https://dev.my-super-app/api/v1/search

    Kibana (логи):
    Dev:  https://dev.my-super-app/kibana
    Test: https://test.my-super-app/kibana

    Kubernetes:
    Dev:  kubectl --context=dev
    Test: kubectl --context=test
    Prod: kubectl --context=prod
```

## Как работает CI/CD

### Основной принцип:

CI/CD реагирует на триггеры (события) и запускает заранее настроенные скрипты.

### Триггеры:

| Событие       | Что запускает                  | Пример                  |
|:--------------|--------------------------------|-------------------------|
| Push в ветку  | Пайплайн сборки и тестов       | Разработчик запушил код |
| Merge Request | Пайплайн проверки перед мержем | Создание MR в GitLab    |
| По расписанию | Ночные тесты, сборка           | Cron каждую ночь        |
| Вручную       | Деплой на стенд                | Кнопка "Deploy" в UI    |
| Тег           | Релизный пайплайн              | git tag v1.0.0          |

### Архитектура CI/CD:

```text
    ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
    │  Developer  │────▶│   Git Repo  │────▶│  CI/CD      │
    │             │     │  (GitLab)   │     │  Server     │
    └─────────────┘     └─────────────┘     └─────────────┘
                                                   │
                        ┌──────────────────────────┼──────────────────────────┐
                        ▼                          ▼                          ▼
                  ┌─────────────┐           ┌─────────────┐           ┌─────────────┐
                  │    Build    │           │    Test     │           │   Deploy    │
                  │   (сборка)  │           │   (тесты)   │           │  (развёрт)  │
                  └─────────────┘           └─────────────┘           └─────────────┘
                        │                          │                          │
                        ▼                          ▼                          ▼
                  ┌─────────────┐           ┌─────────────┐           ┌─────────────┐
                  │ Docker Image│           │   Отчёт     │           │  Kubernetes │
                  │   в Nexus   │           │  SonarQube  │           │    стенд    │
                  └─────────────┘           └─────────────┘           └─────────────┘
```

### Популярные инструменты:

| Инструмент     | Тип                 | Особенности                            |
|:---------------|---------------------|----------------------------------------|
| GitLab CI      | Встроенный в GitLab | Простая настройка через .gitlab-ci.yml |
| Jenkins        | Отдельный сервер    | Гибкий, много плагинов                 |
| GitHub Actions | Встроенный в GitHub | Интеграция с GitHub                    |
| TeamCity       | Отдельный сервер    | От JetBrains                           |
| CircleCI       | Облачный            | Быстрая настройка                      |
| Azure DevOps   | Облачный            | Интеграция с Azure                     |

## Этапы пайплайна

Типичный пайплайн состоит из 3-4 основных этапов:

### Этап 1: Build (Сборка)

**Цель:** Проверить что код компилируется

**Что происходит:**

- GitLab клонирует репозиторий по хешу коммита
- Запускается команда сборки (mvn compile, npm build, etc.)
- Создаётся артефакт (JAR, Docker image, etc.)

**Команды:**

- `mvn compile`
- `npm install && npm run build`
- `docker build -t my-app:version .`

**Если упало:** Код не компилируется, нужно исправить

---

### Этап 2: Test (Тестирование)

**Цель:** Проверить что тесты проходят

**Что происходит:**

- Запускаются unit тесты
- Запускаются интеграционные тесты
- Проверяется покрытие кода (code coverage)
- Статический анализ кода (SonarQube)

**Команды:**
- `mvn test`
- `npm test`
- `sonar-scanner`

**Если упало:** Тесты не проходят или найдены уязвимости

---

### Этап 3: Package (Упаковка)

**Цель:** Создать артефакт для деплоя

**Что происходит:**

- Сборка Docker образа
- Тегирование версии
- Загрузка в реестр (Nexus, Docker Hub)

**Команды:**

- `docker build -t nexus.company.com/my-app:1.0.0 .`
- `docker push nexus.company.com/my-app:1.0.0`

---

### Этап 4: Deploy (Развёртывание)

**Цель:** Развернуть приложение на стенде

**Что происходит:**

- Обновление манифеста в Kubernetes
- K8s скачивает новый образ
- Старая версия удаляется, новая запускается

**Команды:**

- `kubectl set image deployment/my-app my-app=nexus.company.com/my-app:1.0.0`
- `kubectl rollout status deployment/my-app`

**Важно:** Часто запускается вручную по кнопке

---

### Пример пайплайна GitLab CI:

```yaml
    stages:
      - build
      - test
      - deploy

    build:
      stage: build
      script:
        - mvn compile
      artifacts:
        paths:
          - target/*.jar

    test:
      stage: test
      script:
        - mvn test
        - sonar-scanner
      coverage: '/Coverage: \d+\.\d+%/'

    deploy_dev:
      stage: deploy
      script:
        - docker build -t my-app:$CI_COMMIT_SHA .
        - docker push registry/my-app:$CI_COMMIT_SHA
        - kubectl set image deployment/my-app my-app=registry/my-app:$CI_COMMIT_SHA
      environment:
        name: development
      when: manual

    deploy_prod:
      stage: deploy
      script:
        - kubectl set image deployment/my-app my-app=registry/my-app:latest
      environment:
        name: production
      when: manual
      only:
        - main
```

## Интеграция автотестов

### Типы тестов в пайплайне:

| Тип тестов     | Когда запускаются  | Время выполнения | Где                |
|:---------------|--------------------|------------------|--------------------|
| Unit тесты     | При каждом коммите | 1-5 минут        | CI этап            |
| Интеграционные | При каждом коммите | 5-15 минут       | CI этап            |
| Регрессионные  | На test стенде     | 15-60 минут      | CD этап            |
| E2E тесты      | На pre-prod        | 30-120 минут     | CD этап            |
| Нагрузочные    | По расписанию      | 1-4 часа         | Отдельный пайплайн |
| Безопасность   | Перед релизом      | 30-60 минут      | CD этап            |

### Пайплайн с автотестами:

```text
    ┌─────────────┐
    │   Commit    │
    └──────┬──────┘
           ▼
    ┌─────────────┐
    │   Linting   │  ← Проверка стиля кода (ESLint, CheckStyle)
    └──────┬──────┘
           ▼
    ┌─────────────┐
    │ Unit Tests  │  ← Быстрые тесты отдельных модулей
    └──────┬──────┘
           ▼
    ┌─────────────┐
    │Integration  │  ← Тесты взаимодействия компонентов
    │   Tests     │
    └──────┬──────┘
           ▼
    ┌─────────────┐
    │   Deploy    │  ← Деплой на test стенд
    │   to Test   │
    └──────┬──────┘
           ▼
    ┌─────────────┐
    │ Regression  │  ← Регрессионные тесты на стенде
    │   Tests     │
    └──────┬──────┘
           ▼
    ┌─────────────┐
    │   Deploy    │  ← Деплой на pre-prod
    │ to Pre-Prod │
    └──────┬──────┘
           ▼
    ┌─────────────┐
    │    E2E      │  ← End-to-End тесты пользовательских сценариев
    │   Tests     │
    └──────┬──────┘
           ▼
    ┌─────────────┐
    │   Deploy    │  ← Деплой на prod (если все тесты прошли)
    │   to Prod   │
    └─────────────┘
```

### Пример интеграции тестов в GitLab CI:

```yaml
    unit_tests:
      stage: test
      script:
        - mvn test
      rules:
        - if: $CI_PIPELINE_SOURCE == "merge_request_event"

    integration_tests:
      stage: test
      script:
        - mvn verify -Pintegration
      needs: ["unit_tests"]

    deploy_test:
      stage: deploy
      script:
        - ./deploy.sh test
      environment:
        name: test
      when: manual

    regression_tests:
      stage: test
      script:
        - ./run-e2e-tests.sh
      needs: ["deploy_test"]
      rules:
        - if: $CI_COMMIT_BRANCH == "develop"

    e2e_tests:
      stage: test
      script:
        - ./run-e2e-tests.sh --environment pre-prod
      needs: ["deploy_pre_prod"]
      rules:
        - if: $CI_COMMIT_BRANCH == "main"
```

### Рекомендации по организации тестов:

**Быстрые тесты (Unit):**

- Запускать при каждом коммите
- Должны выполняться за 1-5 минут
- Блокируют мерж если упали

**Средние тесты (Integration):**

- Запускать при каждом MR
- Должны выполняться за 5-15 минут
- Проверяют взаимодействие компонентов

**Медленные тесты (E2E, Regression):**

- Запускать на стендах после деплоя
- Могут выполняться 30+ минут
- Можно запускать по расписанию ночью

## Ошибки в пайплайнах

### Частые причины падений:

| Ошибка                | Причина                    | Решение                                    |
|:----------------------|----------------------------|--------------------------------------------|
| Build failed          | Код не компилируется       | Исправить ошибки компиляции                |
| Test failed           | Тесты не проходят          | Исправить код или тесты                    |
| No space on disk      | Кончилось место на runner  | Очистить кэш или обратиться к DevOps       |
| Command not found     | Нет инструмента (mvn, npm) | Проверить Docker image или runner          |
| Timeout               | Пайплайн слишком долгий    | Оптимизировать тесты или увеличить timeout |
| Permission denied     | Нет прав на деплой         | Запросить доступ у DevOps                  |
| Registry auth failed  | Неверные credentials       | Обновить secrets в CI/CD                   |
| K8s deployment failed | Ошибка в манифесте         | Проверить deployment yaml                  |

### Как диагностировать:

1. **Открыть упавший этап** — нажать на красный крестик
2. **Посмотреть логи** — прокрутить до места, где началась ошибка
3. **Найти ошибку** — искать слова ERROR, FAILED, Exception
4. **Воспроизвести локально** — запустить команду у себя в терминале
5. **Исправить и запушить** — новый коммит запустит новый пайплайн

### Пример лога ошибки:

```text
    $ mvn test
    [INFO] Building my-app 1.0.0
    [INFO] Compiling 150 source files
    [ERROR] /src/main/java/UserService.java:[45,12] cannot find symbol
    [ERROR]   symbol:   method getUserById(Long)
    [ERROR]   location: class UserRepository
    [INFO] BUILD FAILURE
```

**Решение:** Метод getUserById не существует в UserRepository

---

### Важные моменты:

- GitLab не даст влить ветку если пайплайн упал
- Любой error блокирует возможность merge
- Нужно запушить патч и пайплайн запустится заново
- Некоторые этапы можно пропускать (через DevOps)

## Best Practices

### Для разработчиков:

- Делайте маленькие коммиты — легче найти ошибку
- Пишите осмысленные сообщения коммитов
- Запускайте тесты локально перед push
- Не коммитьте в main напрямую — используйте MR
- Проверяйте пайплайн после каждого push

### Для тестировщиков:

- Автоматизируйте регрессионные тесты
- Запускайте E2E тесты на pre-prod
- Мониторьте пайплайны после деплоя
- Документируйте тесты
- Синхронизируйте тесты с версией приложения

### Для пайплайнов:

- Keep it fast — пайплайн должен быть быстрым
- Fail fast — останавливайте при первой ошибке
- Кэшируйте зависимости (npm, maven)
- Используйте parallel execution для тестов
- Логируйте всё для отладки

### Для безопасности:

- Не храните секреты в коде
- Используйте CI/CD variables для passwords
- Ограничивайте доступ к prod деплою
- Сканируйте зависимости на уязвимости
- Проверяйте Docker images на security

### Ветвление для CI/CD:

```text
    main ─────────────────────────────▶ (prod деплой)
      ▲                                   │
      │                                   ▼
    develop ──────────────────────────▶ (test деплой)
      ▲                                   │
      │                                   ▼
    feature/* ────────────────────────▶ (dev деплой, manual)
```

### Чеклист перед мержем:

- [ ] Пайплайн зелёный
- [ ] Все тесты прошли
- [ ] Code coverage не упал
- [ ] SonarQube без критических проблем
- [ ] Деплой на test стенд успешен
- [ ] Регрессионные тесты прошли
- [ ] Code review approved

## Глоссарий терминов

- Pipeline - Набор этапов которые выполняются последовательно
- Stage - Этап пайплайна (build, test, deploy)
- Job - Конкретная задача внутри stage
- Runner - Сервер который выполняет job
- Artifact - Файл созданный в пайплайне (JAR, Docker image)
- Trigger - Событие, которое запускает пайплайн
- Manual Job - Job который запускается вручную по кнопке
- Environment - Целевое окружение для деплоя (dev, test, prod)
- Merge Request - Запрос на слияние ветки (GitLab)
- Pull Request - Запрос на слияние ветки (GitHub)
- Rollback - Откат к предыдущей версии приложения
- Blue-Green - Стратегия деплоя с двумя идентичными средами
- Canary - Постепенный деплой на часть пользователей

## Типичный рабочий процесс

### День разработчика с CI/CD:

1. Создать ветку feature/new-feature от develop
2. Написать код и тесты
3. Запустить тесты локально
4. Сделать commit и push
5. GitLab автоматически запускает пайплайн
6. Ждать пока пайплайн пройдёт (5-15 минут)
7. Если упал — исправить и снова push
8. Если прошёл — создать Merge Request
9. Ждать code review
10. После approval — мерж в develop
11. Автоматический деплой на test стенд
12. Тестировщики проверяют функционал
13. После тестов — мерж в main
14. Автоматический деплой на prod

### День тестировщика с CI/CD:

1. Получить уведомление о новом деплое на test
2. Проверить changelog что нового
3. Запустить регрессионные автотесты
4. Проверить отчёт о тестах
5. Если тесты упали — создать баг разработчику
6. Если тесты прошли — ручное тестирование новых фич
7. Approval для деплоя на pre-prod
8. Финальное тестирование на pre-prod
9. Approval для релиза на prod

## Сравнение подходов

### Без CI/CD vs С CI/CD:

| Метрика                   | Без CI/CD   | С CI/CD              |
|:--------------------------|-------------|----------------------|
| Время до обнаружения бага | Недели      | Минуты               |
| Частота релизов           | Раз в месяц | Несколько раз в день |
| Время на деплой           | Часы        | Минуты               |
| Ручная работа             | Много       | Минимум              |
| Стресс при релизе         | Высокий     | Низкий               |
| Откат при проблеме        | Сложный     | Быстрый              |
| Доверие к коду            | Низкое      | Высокое              |

## Ключевые выводы

1. **CI/CD это не инструмент** — это практика и культура разработки
2. **Автоматизируйте всё** — что можно автоматизировать, должно быть автоматизировано
3. **Тесты это часть кода** — без тестов нет CI/CD
4. **Быстрая обратная связь** — разработчик должен знать о проблемах через минуты
5. **Деплой должен быть скучным** — если деплой это событие, что-то не так
6. **Мониторьте пайплайны** — время выполнения, процент успешных, частые ошибки
7. **Постоянно улучшайте** — оптимизируйте медленные этапы, добавляйте новые тесты
