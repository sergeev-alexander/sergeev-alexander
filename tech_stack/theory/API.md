  </summary>

Для построения API, можно выделить несколько ключевых архитектурных стилей, выбор между которыми
зависит от контекста: требований к производительности, связности сервисов и типа клиентов.

Самый распространённый - `REST`. Это архитектурный стиль, основанный на ресурсах, URI и стандартных
HTTP-методах. Он stateless, кэшируемый и идеально подходит для публичных веб-API, мобильных бэкендов и CRUD-сервисов. В
Java-экосистеме он отлично поддерживается через Spring Boot, JAX-RS. Главный компромисс - текстовые форматы вроде JSON
добавляют overhead, но дают простоту отладки и широкую совместимость.

Если нужна высокая производительность и строгий контракт - например, для внутреннего взаимодействия микросервисов - я
рассматриваю `RPC`, в частности `gRPC`. Здесь контракт описывается в IDL-файлах, используется бинарная сериализация
Protocol
Buffers и HTTP/2. Это даёт низкую задержку и типобезопасность, но требует более сложной инфраструктуры и менее
универсален для внешних клиентов.

Когда фронтенду нужна гибкость в получении данных - чтобы избежать over- или under-fetching - следует использовать
`GraphQL`. Клиент сам описывает, какие поля ему нужны, а сервер возвращает ровно эту структуру. Это мощно для сложных
UI,
но добавляет сложность в кеширование, валидацию и защиту от сложных запросов.

Для асинхронных, слабо связанных систем используется event-driven подход: Kafka, RabbitMQ. Сервисы обмениваются
событиями
через брокер, что обеспечивает отказоустойчивость, масштабируемость и поддержку паттернов вроде CQRS или Saga. Здесь
важен выбор формата событий - Avro, Protobuf - и стратегия версионирования.

Остальные стили решают точечные задачи:

- `SOAP` - для legacy enterprise-интеграций с требованиями к транзакциям.
- `Webhooks` - когда внешний провайдер должен уведомлять нас о событиях.
- `WebSockets` и `SSE` (`Server-Sent Events`) - для real-time сценариев вроде чатов или
  живых уведомлений.

Итог:

Если нет явных требований к производительности или гибкости данных - `REST`.
Если нужна скорость и строгость контракта внутри кластера - `gRPC`.
Если фронтенд диктует сложную структуру данных - `GraphQL`.
Если система должна быть отказоустойчивой и масштабируемой - `Event-driven` подход (Брокеры сообщений).

> API (Application Programming Interface) - это контракт между различными программными компонентами, который определяет,
> как они могут взаимодействовать.
>
> Он описывает набор доступных методов и данных, которые одна программа может запросить у другой, не вдаваясь в детали
> её
> внутренней реализации.

## API Architectural Styles

Стиль проектирования API - это набор ограничений и принципов, определяющих, как компоненты системы взаимодействуют друг
с другом через интерфейсы.

Стиль задаёт:

- как передаются запросы и ответы,
- как идентифицируются ресурсы,
- является ли взаимодействие синхронным или асинхронным,
- как обрабатываются ошибки,
- какие форматы данных используются и т.д.

## REST (Representational State Transfer)

- Архитектурные принципы:
    - Архитектурный стиль, основанный на ограничениях Роя Филдинга: client-server, stateless, cacheable, uniform
      interface, layered system, code-on-demand (опционально)
    - Опирается на ресурсы, идентифицируемые URI, и стандартные HTTP-методы
- Модель взаимодействия:
    - Синхронный запрос–ответ (request–response)
    - Клиент инициирует взаимодействие
- Контракт / Спецификация:
    - Неформальный или частично формализованный (OpenAPI/Swagger)
    - Нет строгого контракта времени компиляции
- Тип данных и сериализация:
    - Обычно JSON или XML (текстовые, человекочитаемые)
- Поддержка в Java-экосистеме:
    - Spring Boot (WebMvc/WebFlux), JAX-RS (Jersey, RESTEasy), Micronaut, Quarkus
- Типичные сценарии использования:
    - Публичные веб-API, мобильные бэкенды, CRUD-сервисы, интеграции с фронтендом
- Производительность:
    - Средняя (текстовые форматы, HTTP overhead)
- Безопасность:
    - HTTPS, OAuth 2.0, JWT, API Keys

---

## RPC (Remote Procedure Call)

- Архитектурные принципы:
    - Клиент вызывает функцию/метод на удалённом сервере как будто она локальная
    - Стиль фокусируется на операциях, а не на ресурсах
    - Часто нарушает uniform interface REST
- Модель взаимодействия:
    - Синхронный или асинхронный запрос–ответ
    - Может поддерживать streaming (в современных реализациях)
- Контракт / Спецификация:
    - Строгий контракт: определяется в IDL (Interface Definition Language)
    - Примеры: .proto (Protocol Buffers) или .thrift
- Тип данных и сериализация:
    - Бинарные форматы: Protocol Buffers (gRPC), Apache Thrift, или JSON (в JSON-RPC)
- Поддержка в Java-экосистеме:
    - gRPC-Java, Spring gRPC, Apache Thrift, JSON-RPC (библиотеки вроде jsonrpc4j)
- Типичные сценарии использования:
    - Внутренние микросервисы с высокой нагрузкой
    - Low-latency системы, когда важна производительность и чёткость контракта
- Производительность:
    - Высокая (бинарные форматы, HTTP/2)
- Безопасность:
    - TLS, mTLS, токены аутентификации

## GraphQL

- Архитектурные принципы:
    - Запрос-ориентированный стиль: клиент отправляет запрос с описанием желаемой структуры данных, сервер возвращает
      ровно это
    - Инверсия контроля - сервер предоставляет "граф возможностей", клиент выбирает, что запрашивать
- Модель взаимодействия:
    - Синхронный запрос–ответ (обычно поверх HTTP POST)
    - Поддерживает подписки (subscriptions) через WebSocket (асинхронно)
- Контракт / Спецификация:
    - Схема типов (GraphQL Schema) - строгая, но динамически исполняемая
    - Контракт проверяется при выполнении запроса
- Тип данных и сериализация:
    - JSON (всегда). Запрос и ответ - структурированные JSON-объекты
- Поддержка в Java-экосистеме:
    - DGS (Netflix), GraphQL Java, Spring for GraphQL, Quarkus GraphQL
- Типичные сценарии использования:
    - Фронтенды с разнообразными требованиями к данным (web/mobile)
    - Избегание over-fetching/under-fetching данных
- Производительность:
    - Средняя (риск N+1 проблемы, сложность кеширования)
- Безопасность:
    - HTTPS, авторизация на уровне полей, rate limiting

## Event-driven / Messaging

- Архитектурные принципы:
    - Взаимодействие через асинхронную передачу событий
    - Основано на паттернах Pub/Sub или Message Queue
    - Сервисы слабо связаны (loose coupling), часто используют CQRS или Event Sourcing
- Модель взаимодействия:
    - Асинхронное, однонаправленное. Нет немедленного ответа
    - Возможны retry, dead-letter queues, ordering guarantees
- Контракт / Спецификация:
    - Контракт - событийная схема (Avro, JSON Schema, Protobuf)
    - Часто версионируется
- Тип данных и сериализация:
    - JSON, Avro, Protobuf, XML - в зависимости от брокера и требований
- Поддержка в Java-экосистеме:
    - Spring Boot + Kafka/RabbitMQ, Quarkus + SmallRye, Project Reactor, Vert.x, Jakarta Messaging (JMS)
- Типичные сценарии использования:
    - Микросервисы с высокой отказоустойчивостью
    - Обработка в реальном времени, аудит, аналитика, Saga-транзакции
- Производительность:
    - Высокая (асинхронность, батчинг)
- Безопасность:
    - Шифрование сообщений, SASL, ACL брокера

## SOAP (Simple Object Access Protocol)

- Архитектурные принципы:
    - Строгий, стандартизированный протокол (не стиль), основанный на XML и WSDL
    - Поддерживает WS-* расширения (безопасность, транзакции, надёжная доставка)
- Модель взаимодействия:
    - Синхронный запрос–ответ (иногда one-way)
- Контракт / Спецификация:
    - WSDL - машинно-читаемый формальный контракт
    - Контракт создаётся до реализации (contract-first)
- Тип данных и сериализация:
    - XML (тяжёлый, многословный)
- Поддержка в Java-экосистеме:
    - Apache CXF, Spring Web Services, JAX-WS (встроено до Java 11), Metro
- Типичные сценарии использования:
    - Legacy enterprise-интеграции (банки, госуслуги)
    - Где требуется формальная спецификация, ACID-транзакции на уровне протокола, или WS-Security
- Производительность:
    - Низкая (XML overhead, сложность парсинга)
- Безопасность:
    - WS-Security, WS-Trust, цифровые подписи

## Webhooks

- Архитектурные принципы:
    - Push-based механизм: сервер уведомляет клиент, отправляя HTTP-запрос на его endpoint при наступлении события
    - Это обратный вызов (callback) по HTTP
- Модель взаимодействия:
    - Асинхронный, но запрос инициирует сервер
    - Клиент должен предоставить публичный endpoint
- Контракт / Спецификация:
    - Неформальный. Часто задокументирован в виде JSON-примеров
- Тип данных и сериализация:
    - Обычно JSON, иногда form-encoded
- Поддержка в Java-экосистеме:
    - Реализуется вручную через Spring Web (как обычный контроллер) + HTTP-клиент (OkHttp, WebClient) для отправки
- Типичные сценарии использования:
    - Интеграции с внешними провайдерами (Stripe, GitHub, Slack)
    - Когда polling неэффективен
- Производительность:
    - Высокая (push-модель, нет polling)
- Безопасность:
    - HTTPS, подписи HMAC, IP whitelist

---

## WebSockets

- Архитектурные принципы:
    - Полнодуплексная связь поверх TCP
    - Постоянное соединение между клиентом и сервером для real-time коммуникации
- Модель взаимодействия:
    - Асинхронный, двунаправленный
    - Сервер и клиент могут инициировать отправку сообщений в любое время
- Контракт / Спецификация:
    - Неформальный. Обычно определяется протоколом приложения поверх WebSocket
- Тип данных и сериализация:
    - JSON, бинарные данные, текст - любые форматы
- Поддержка в Java-экосистеме:
    - Spring WebSocket, Java-WebSocket, Netty, Vert.x
- Типичные сценарии использования:
    - Real-time приложения: чаты, игры, live-обновления, торговые терминалы
- Производительность:
    - Очень высокая (минимальный overhead после handshake)
- Безопасность:
    - WSS (WebSocket Secure), токены в заголовках

---

## Server-Sent Events (SSE)

- Архитектурные принципы:
    - Однонаправленный поток данных от сервера к клиенту поверх HTTP
    - Сервер "толкает" обновления клиенту в real-time
- Модель взаимодействия:
    - Асинхронный, однонаправленный (только сервер → клиент)
    - Автоматическое переподключение
- Контракт / Спецификация:
    - Стандарт W3C. Простой текстовый протокол с полями data, event, id, retry
- Тип данных и сериализация:
    - Текстовые данные (обычно JSON в поле data)
- Поддержка в Java-экосистеме:
    - Spring WebFlux, SseEmitter, Reactor, RxJava
- Типичные сценарии использования:
    - Live-обновления UI, уведомления, мониторинг, новостные ленты
- Производительность:
    - Высокая (HTTP keep-alive, меньше overhead чем polling)
- Безопасность:
    - HTTPS, CORS, авторизация через query params или headers
